# ADR-004: Flutter for Cross-Platform Client

## Status
Accepted

## Date
2026-03-20

## Context

The restaurant POS platform requires client applications for multiple form
factors and use cases:

- **POS Terminal**: Primary order-taking interface on Android tablets (10-inch)
- **Waiter Terminal**: Handheld order-taking on smaller Android devices (8-inch)
- **Kitchen Display System (KDS)**: Order display on wall-mounted screens
- **Self-Service Kiosk**: Customer-facing order interface on large touchscreens
- **Web Dashboard**: Management interface for reporting and configuration
- **Possible future**: iOS tablets for premium market segment

All these clients share significant business logic:
- Order model and calculations (taxes, discounts, modifiers)
- Money handling (integer cents, rounding)
- Sync protocol with Cloud Hub and LAN peers
- Authentication and authorization
- Local database operations

Building separate native applications for each platform would multiply
development effort across a team of 1-5 developers.

## Decision

We adopt **Flutter 3.x with Dart** as the cross-platform framework for all
client applications.

### Platform Coverage

| Application       | Platform          | Priority | Timeline  |
|--------------------|-------------------|----------|-----------|
| POS Terminal      | Android tablet    | P0       | Phase 1   |
| Waiter Terminal   | Android handheld  | P0       | Phase 1   |
| KDS               | Android / Web     | P1       | Phase 2   |
| Self-Service Kiosk| Android / Web     | P2       | Phase 5   |
| Web Dashboard     | Web (Flutter Web) | P1       | Phase 3   |
| iOS (if needed)   | iOS tablet        | P3       | Phase 8+  |

### Architecture

```
lib/
  core/               -- Shared business logic (platform-independent)
    models/            -- Domain entities (Order, Product, Payment, etc.)
    services/          -- Business logic services
    sync/              -- Sync engine (LAN + Cloud)
    database/          -- Drift ORM schemas, DAOs, migrations
    money/             -- Money type and calculations
  features/            -- Feature modules
    pos/               -- POS terminal UI and logic
    kds/               -- Kitchen display UI and logic
    kiosk/             -- Self-service kiosk UI
    dashboard/         -- Web management dashboard
  shared/              -- Shared UI components
    widgets/           -- Custom POS widgets (order panel, keypad, etc.)
    themes/            -- Theming and branding
    l10n/              -- Localization (DE, FR, IT, EN)
```

### Key Technical Choices Within Flutter

- **State Management**: Riverpod for dependency injection and state management.
  Chosen over Bloc (too much boilerplate for POS use cases) and Provider
  (Riverpod is the evolution with better testing support).
- **Local Database**: Drift ORM for SQLite (see ADR-006). Type-safe queries,
  reactive streams, migration support.
- **Networking**: Dio for HTTP client (interceptors, retry logic, logging).
- **Printing**: ESC/POS protocol via TCP sockets for receipt and kitchen
  printers.
- **Navigation**: GoRouter for declarative routing across all platforms.

### POS-Specific UI Requirements

Flutter's rendering engine (Skia/Impeller) provides pixel-perfect control
over the UI, which is critical for POS applications:

- **Split-screen layouts**: Order panel on right, menu grid on left, with
  drag-and-drop between sections.
- **Large touch targets**: Minimum 48dp for speed during rush hours.
- **Custom keypad**: Numeric input for quantities, prices, and PLU codes.
- **Gesture navigation**: Swipe between table views, pinch-to-zoom floor
  plan.
- **Real-time updates**: Order status changes, KDS ticket updates, table
  status -- all reactive via Drift streams.
- **Kiosk mode**: Full-screen, no system navigation, locked to POS app.

## Consequences

### Positive

- **Single codebase**: One codebase for Android, web, and potentially iOS.
  Business logic written once, tested once.
- **Rapid iteration**: Hot reload during development. Change UI, see result
  in under a second.
- **Consistent UX**: Every platform renders identically because Flutter
  owns the entire rendering pipeline (not using native widgets).
- **Drift ORM**: Best-in-class SQLite integration for Flutter. Type-safe
  queries, automatic migration generation, reactive `watch()` queries.
- **Strong typing**: Dart's sound null safety catches errors at compile
  time. Critical for financial calculations.
- **Custom rendering**: Flutter's canvas gives us full control over custom
  POS UI elements (floor plan editor, order timeline, receipt preview).

### Negative

- **Dart ecosystem**: Smaller package ecosystem compared to JavaScript or
  Kotlin. Some integrations may require writing platform-specific plugins.
- **Flutter Web performance**: Flutter Web (via CanvasKit) produces larger
  initial bundle sizes and slower first-paint compared to React/Vue.
  Acceptable for a management dashboard, but not ideal for a public-facing
  website.
- **Android-specific APIs**: Some device-specific features (NFC, Bluetooth
  printers, payment terminals) still require platform channel bridges to
  native code.
- **Team hiring**: Fewer Dart/Flutter developers in the market compared to
  React Native or native Android/iOS developers.

### Risks

- **Flutter Web maturity**: Flutter Web is less mature than Flutter mobile.
  Complex web-specific features (deep linking, SEO, browser APIs) may
  require workarounds. Mitigation: the web dashboard is an internal tool,
  not a public-facing site. Performance and SEO are not critical.
- **Framework lock-in**: If Flutter's development slows or Google
  deprioritizes it, migration would be costly. Mitigation: business logic
  is in pure Dart with minimal Flutter dependencies. UI is the coupled
  layer, but this is true of any framework.
- **Plugin maintenance**: Critical plugins (Drift, Riverpod, Dio) are
  community-maintained. Mitigation: all chosen plugins are actively
  maintained with strong communities. Drift is exceptionally well-maintained
  by Simon Binder.

## Alternatives Considered

### Native Android (Kotlin) + Separate Web (React)

Build the POS natively in Kotlin for Android, and a separate React
application for the web dashboard.

**Rejected because:**
- Two codebases means duplicated business logic (order calculations,
  sync protocol, money handling).
- Two tech stacks to maintain, test, and deploy.
- Team of 1-5 cannot effectively maintain both.
- Adding iOS later would require a third codebase.

### React Native

Cross-platform framework using JavaScript/TypeScript and native components.

**Rejected because:**
- React Native uses native platform widgets, which means the POS UI
  would look and behave differently on Android vs. web vs. iOS. Flutter's
  custom rendering engine gives us identical behavior everywhere.
- JavaScript bridge performance overhead for computationally intensive
  POS operations (real-time order calculations, sync processing).
- SQLite integration in React Native is less mature than Drift for
  Flutter. No equivalent to Drift's type-safe, reactive query system.
- React Native's "new architecture" (Fabric, TurboModules) is still
  stabilizing. Flutter's rendering pipeline is more mature.

### Kotlin Multiplatform (KMP)

Shared business logic in Kotlin with platform-specific UI (Jetpack
Compose for Android, SwiftUI for iOS, Compose for Web).

**Rejected because:**
- Compose for Web is experimental and not production-ready.
- Still requires platform-specific UI code for each target.
- Smaller ecosystem and less community tooling compared to Flutter.
- Learning curve for the team, which has existing Dart/Flutter experience.

### Electron / Tauri for Desktop/Web + Native Mobile

Desktop app via Electron/Tauri with a separate native mobile app.

**Rejected because:**
- Our primary target is Android tablets, not desktop.
- Running Electron on a restaurant tablet wastes resources.
- Still requires a separate mobile codebase.
