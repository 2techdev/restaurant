# ADR-011: Feature Flags for License Tier Enforcement

## Status
Accepted

## Date
2026-03-20

## Context

The restaurant POS platform is a commercial product with multiple subscription
tiers. Different customers pay for different levels of functionality:

- **Starter**: Single branch, limited devices, basic POS features
- **Professional**: Multi-branch, more devices, KDS, reporting
- **Enterprise**: Unlimited branches, API access, custom integrations,
  white-labeling

Additionally, some features are sold as add-ons:
- QR ordering module
- Self-service kiosk module
- ERPNext integration
- Advanced analytics
- Multi-language menu support

The platform must enforce these license boundaries at runtime -- both in
the Flutter client (hiding unavailable features from the UI) and in the
Cloud Hub (rejecting API calls for features the tenant has not licensed).

Key constraints:
- **Offline enforcement**: The Branch Runtime operates offline (ADR-002).
  License checks cannot depend on real-time cloud communication.
- **Tamper resistance**: A technically savvy operator should not be able
  to easily unlock features by modifying local data.
- **Graceful handling**: Accessing an unlicensed feature should produce
  a clear message, not a crash or undefined behavior.

## Decision

We implement license enforcement via **feature flags embedded in a
cryptographically signed license token**, checked at runtime before
module initialization.

### License Token Structure

The license token is a JWT (JSON Web Token) signed with an asymmetric key
pair (Ed25519). The Cloud Hub holds the private key; clients verify with
the public key embedded in the application binary.

```json
{
  "iss": "2tech-license",
  "sub": "tenant-uuid-here",
  "iat": 1710892800,
  "exp": 1713571200,
  "tier": "professional",
  "features": {
    "pos_basic": true,
    "pos_advanced": true,
    "kds": true,
    "kiosk": false,
    "qr_ordering": true,
    "web_ordering": false,
    "erpnext_bridge": false,
    "advanced_analytics": true,
    "multi_language_menu": true,
    "api_access": false,
    "white_label": false
  },
  "limits": {
    "max_branches": 5,
    "max_devices_per_branch": 8,
    "max_products": 500,
    "max_users": 25
  },
  "grace_period_days": 7
}
```

### Token Lifecycle

```
Cloud Hub (Go)
  |
  | 1. Tenant subscribes / changes plan
  | 2. Cloud Hub generates new license token
  | 3. Token signed with Ed25519 private key
  | 4. Token stored in tenant record (PostgreSQL)
  |
  v
Branch Runtime (Flutter)
  |
  | 5. Device syncs and receives new token
  | 6. Token verified with embedded public key
  | 7. Token cached in encrypted local storage (SQLCipher)
  | 8. Feature flags extracted and applied
  |
  v
Runtime Enforcement
  |
  |-- UI Layer: Hide/show features based on flags
  |-- Service Layer: Check flags before business logic
  |-- API Layer (Cloud): Reject calls for disabled features
```

### Enforcement Points

**Flutter Client (UI Enforcement):**

```dart
class FeatureGate extends ConsumerWidget {
  final String featureKey;
  final Widget child;
  final Widget? placeholder;

  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final license = ref.watch(licenseProvider);
    if (license.isFeatureEnabled(featureKey)) {
      return child;
    }
    return placeholder ?? UpgradePromptWidget(feature: featureKey);
  }
}
```

Usage in UI:
```dart
FeatureGate(
  featureKey: 'kds',
  child: KitchenDisplayScreen(),
  placeholder: FeatureLockedCard(
    title: 'Kitchen Display System',
    description: 'Upgrade to Professional to unlock KDS',
  ),
)
```

**Flutter Client (Service Enforcement):**

```dart
class OrderService {
  Future<Order> createOrder(CreateOrderCommand cmd) async {
    final license = _licenseService.currentLicense;

    // Check numeric limit
    if (cmd.channel == OrderChannel.kiosk &&
        !license.isFeatureEnabled('kiosk')) {
      throw FeatureNotLicensedException('kiosk');
    }

    // Proceed with order creation
    return _orderEngine.create(cmd);
  }
}
```

**Cloud Hub (API Enforcement):**

```go
func (m *LicenseMiddleware) CheckFeature(feature string) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            tenant := TenantFromContext(r.Context())
            if !tenant.License.HasFeature(feature) {
                http.Error(w, "Feature not licensed: "+feature, http.StatusForbidden)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}
```

### Offline License Validation

When the device is offline:

1. The cached license token is verified using the embedded Ed25519 public
   key. No network call needed.
2. Token expiration is checked against the device clock.
3. If the token is expired but within the `grace_period_days`, full
   functionality continues with a warning banner.
4. If the token is expired beyond the grace period, the system enters
   **grace mode**: basic POS functionality works (taking orders, cash
   payments), but advanced features are locked.
5. On connectivity restoration, the device requests a fresh token from
   the Cloud Hub.

### Grace Mode (Expired License)

Grace mode ensures a restaurant can always operate at a basic level, even
if billing issues prevent license renewal:

- Order creation: **allowed** (basic POS)
- Cash payments: **allowed**
- Receipt printing: **allowed**
- Kitchen display: **allowed** (safety -- food must be prepared)
- Card payments: **depends on payment terminal** (independent of our license)
- Reporting: **locked**
- Kiosk / QR ordering: **locked**
- New device registration: **locked**

## Consequences

### Positive

- **Offline enforcement**: License checks work without internet via
  cryptographic verification of the cached token.
- **Tamper resistance**: Ed25519 signature prevents modification of
  feature flags. The public key is embedded in the compiled binary.
- **Centralized management**: License changes propagate automatically
  via sync. Upgrading a tenant is a Cloud Hub operation, not a device
  reconfiguration.
- **Graceful degradation**: Grace mode ensures restaurants can always
  serve customers, even during billing disputes.
- **Clean module boundaries**: Feature flags align with module boundaries
  (ADR-001). Each module can check its own feature flag at initialization.
- **Analytics**: Feature flag checks can be logged to understand which
  features are used, informing product decisions.

### Negative

- **Clock manipulation**: A user could set the device clock back to
  extend an expired token. Mitigation: track clock skew detection
  (compare device time with last known server time from sync).
- **Binary patching**: A determined attacker could patch the public key
  or bypass the check. Mitigation: this is acceptable for a SaaS
  business (not DRM). The value is in the service, not just the software.
- **Token refresh coordination**: When a plan changes, all devices for
  that tenant need to receive the new token. This depends on sync
  (ADR-014).
- **Feature flag granularity**: Adding too many fine-grained flags
  increases complexity. Must resist feature-flag proliferation.

### Risks

- **Grace mode abuse**: A tenant could intentionally let their license
  expire to use basic POS for free. Mitigation: grace period is limited
  (7 days). After that, the system shows prominent renewal prompts.
  Cloud features (reporting, sync) stop working immediately.
- **Token size growth**: As features are added, the JWT payload grows.
  Mitigation: use short feature keys, consider using a bitfield instead
  of named booleans if the list exceeds 50 features.
- **Split-brain license**: If the cloud updates the license but the
  device has a cached old token, the device may enforce outdated rules.
  Mitigation: license refresh is high-priority in the sync queue. Devices
  check for updated tokens on every successful cloud connection.

## Alternatives Considered

### Server-Side License Check on Every Operation

Call the Cloud Hub to verify the license before every feature access.

**Rejected because:**
- Violates offline-first principle (ADR-002). No internet means no
  license check, which means no feature access.
- Adds latency to every operation.
- Cloud Hub becomes a single point of failure for all branch operations.

### Obfuscated Local Configuration File

Store feature flags in an encrypted local configuration file without
cryptographic signing.

**Rejected because:**
- Without asymmetric signing, a sophisticated user could reverse-engineer
  the encryption key and modify the flags.
- No centralized management -- changing a license would require
  generating and distributing new config files.
- JWT-based approach is industry-standard and well-tooled.

### Hardware Dongle / USB License Key

Physical license key connected to the POS device.

**Rejected because:**
- Additional hardware cost per device.
- USB port availability on tablets is limited.
- Lost or damaged dongles prevent restaurant operation.
- Does not scale well for multi-device branches.
- Physical distribution logistics for license changes.

### App Store In-App Purchase for Feature Unlocking

Use Google Play / App Store in-app purchase mechanisms.

**Rejected because:**
- 15-30% platform commission on license fees.
- Platform dependency for license management.
- Does not work for web or self-hosted deployments.
- IAP APIs are designed for consumer apps, not B2B SaaS.
- No support for multi-branch, multi-device licensing.
