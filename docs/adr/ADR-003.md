# ADR-003: ERPNext as Accounting Bridge, Not as Core

## Status
Accepted

## Date
2026-03-20

## Context

A restaurant POS platform needs more than just order-taking. It needs:

- **Accounting**: Revenue recognition, expense tracking, tax reports, P&L
- **Stock management**: Ingredient tracking, purchase orders, suppliers
- **Tax compliance**: VAT returns, country-specific fiscal reporting
- **Invoicing**: Customer invoices, supplier invoice matching
- **HR basics**: Employee records (for payroll integration)

Building all of this from scratch would take years and distract from the
core product (the POS experience). We need an existing back-office system
that handles these concerns.

ERPNext (Community Edition) is a full-featured open-source ERP that covers
accounting, stock, HR, and more. It is built on the Frappe framework
(Python/MariaDB) and has a web-based UI.

The question is: **How should ERPNext fit into our architecture?**

Two approaches were considered:

1. **ERPNext as the core**: Build the POS as an ERPNext module. Use Frappe's
   UI framework. Store all data in ERPNext's MariaDB. The POS is an extension
   of ERPNext.
2. **ERPNext as a bridge**: Build the POS independently with its own runtime
   and data model. ERPNext receives summarized data (daily sales, stock
   movements, invoices) via API integration. ERPNext handles back-office
   concerns; the POS handles front-of-house.

## Decision

We adopt ERPNext Community Edition as a **back-office accounting bridge**,
not as the core system.

### Integration Architecture

```
Branch Runtime (Flutter/SQLite)
    |
    v
Cloud Hub (Go/PostgreSQL)  -----> ERPNext Bridge Module
    |                                    |
    |                                    v
    |                              ERPNext (Frappe/MariaDB)
    |                                    |
    |                                    +--> Accounting (GL entries)
    |                                    +--> Stock (inventory movements)
    |                                    +--> Tax Reports (VAT returns)
    |                                    +--> Invoicing (customer/supplier)
    |                                    +--> HR (employee records)
    v
 Reporting & Analytics
```

### What Flows to ERPNext

| Data                  | Frequency      | Direction          |
|-----------------------|----------------|--------------------|
| Daily sales summary   | End of day     | Cloud -> ERPNext   |
| Payment reconciliation| End of day     | Cloud -> ERPNext   |
| Stock consumption     | End of day     | Cloud -> ERPNext   |
| Purchase orders       | On creation    | ERPNext -> Cloud   |
| Supplier invoices     | On receipt     | ERPNext -> Cloud   |
| Product catalog       | On change      | ERPNext <-> Cloud  |
| Employee records      | On change      | ERPNext -> Cloud   |
| GL journal entries    | End of day     | Cloud -> ERPNext   |
| Tax period reports    | Monthly        | ERPNext (internal) |

### What Stays in Our System

- Real-time order management
- Kitchen display routing
- Table and floor management
- Payment terminal interaction
- Receipt generation and printing
- Device management and sync
- Shift management
- Customer-facing displays
- All offline operations

### Bridge Module Design

The ERPNext Bridge is a module in the Cloud Hub that:

1. **Translates** our domain models to ERPNext doctypes (e.g., our `Order`
   becomes an ERPNext `Sales Invoice`).
2. **Batches** operations for efficiency (one API call per end-of-day
   summary, not per order).
3. **Handles failures** gracefully -- if ERPNext is down, data queues
   locally and retries.
4. **Is optional** -- the POS platform functions fully without ERPNext.
   ERPNext is enabled per tenant as an add-on.

### ERPNext API Integration

We use ERPNext's REST API (Frappe API) for all integration:
- Authentication via API keys (per tenant)
- Standard CRUD operations on doctypes
- Custom server scripts in ERPNext for specialized logic
- Webhook callbacks for ERPNext-initiated changes

## Consequences

### Positive

- **Full UX control**: Our POS UI is purpose-built for restaurant workflows.
  We are not constrained by Frappe's web UI framework or its page-based
  navigation model.
- **Offline-first preserved**: ERPNext's web-based architecture cannot work
  offline. By keeping it as a bridge, our offline-first design is unaffected.
- **Performance**: The POS runtime never waits for ERPNext. All ERPNext
  integration is asynchronous and batched.
- **Flexibility**: We can swap ERPNext for another accounting system (Bexio,
  DATEV, Sage) by building a different bridge adapter. The POS core does
  not change.
- **License clarity**: ERPNext Community is GPL v3. By treating it as a
  separate service accessed via API, we avoid GPL contamination of our
  proprietary POS codebase.
- **Selective adoption**: Not every customer needs full ERP. Smaller
  restaurants can use the POS without ERPNext and handle accounting
  manually or via CSV exports.

### Negative

- **Integration maintenance**: We must build and maintain the bridge module,
  handle ERPNext API changes across versions, and test the integration.
- **Data duplication**: Products, customers, and employees exist in both
  systems. We must keep them synchronized and handle conflicts.
- **ERPNext expertise needed**: Someone on the team must understand ERPNext
  doctypes, workflows, and customization to build and maintain the bridge.
- **Two systems to operate**: Customers using ERPNext must run both our
  Cloud Hub and an ERPNext instance (or we host it for them).

### Risks

- **ERPNext API instability**: Frappe/ERPNext API can change between major
  versions. Mitigation: pin ERPNext version per tenant, test bridge against
  specific versions, use API versioning where available.
- **Sync drift**: If the bridge fails silently, accounting data diverges
  from POS data. Mitigation: reconciliation checks, monitoring, daily
  summary verification.
- **ERPNext Community limitations**: Some features may only be in ERPNext
  paid editions. Mitigation: evaluate CE capabilities upfront, build
  custom doctypes where needed.

## Alternatives Considered

### ERPNext as Core System

Build the entire POS as an ERPNext app/module using the Frappe framework.

**Rejected because:**
- **No offline support**: Frappe is a web framework. It requires a running
  server and database connection. There is no viable path to offline-first
  operation on Android tablets.
- **UI limitations**: Frappe's UI framework is designed for back-office data
  entry, not for high-speed POS interactions (large touch targets, split
  screen, gesture navigation, real-time KDS).
- **Performance**: Frappe/Python/MariaDB stack is not optimized for the
  real-time, low-latency requirements of a POS system processing hundreds
  of orders per hour.
- **GPL v3 license**: ERPNext Community is GPL. Building the POS as an
  ERPNext module would make our entire codebase GPL, preventing proprietary
  licensing.
- **Deployment complexity**: ERPNext requires MariaDB, Redis, Gunicorn,
  Nginx, and a bench environment. This is too heavy for branch-level
  deployment and too complex for self-hosted customers.

### Build Accounting from Scratch

Implement our own general ledger, chart of accounts, tax calculation,
and financial reporting.

**Rejected because:**
- Accounting software is a regulated, complex domain. Getting double-entry
  bookkeeping, tax compliance, and financial reporting right takes years
  of development and domain expertise.
- We would be building a product outside our core competency (restaurant
  POS).
- ERPNext provides a battle-tested, localized (Swiss, German) accounting
  system that we can leverage immediately.

### Third-Party SaaS Accounting (Bexio, Xero, DATEV)

Integrate with a commercial SaaS accounting provider instead of ERPNext.

**Not rejected outright** -- this is a future option. However, ERPNext was
chosen as the initial bridge because:
- Open source: no per-seat licensing costs for customers
- Self-hostable: data sovereignty for European customers
- Full feature set: accounting + stock + HR in one system
- We can add Bexio/DATEV adapters later as alternative bridges
