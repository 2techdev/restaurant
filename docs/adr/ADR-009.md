# ADR-009: Single Order Engine with Channel Adapters

## Status
Accepted

## Date
2026-03-20

## Context

The restaurant POS platform supports multiple ordering channels:

- **POS Terminal**: Staff takes order at the register or counter
- **Waiter Terminal**: Staff takes order at the table with handheld device
- **QR Code (Guest Order)**: Guest scans QR code at table, orders from phone
- **Self-Service Kiosk**: Guest orders at a standing kiosk terminal
- **Web Ordering**: Guest orders online for takeaway/delivery
- **Third-Party Delivery**: Orders from Uber Eats, Lieferando, etc. (future)

Each channel has different UI requirements, different input methods, and
different validation rules. However, once an order enters the system, it
must be treated identically:

- The kitchen sees one unified order queue regardless of source
- Payment processing is the same regardless of how the order was placed
- Tax calculation, discounts, and pricing must be consistent
- Reporting aggregates all channels
- Fiscal signing treats all orders equally

The risk of building separate order systems per channel is significant:
duplication of business logic, inconsistent behavior, and maintenance
overhead that scales with the number of channels.

## Decision

We implement a **single Order Engine** at the core of the system, with
**channel-specific adapters** that handle input transformation and
channel-specific UX.

### Architecture

```
Channel Adapters (Input Layer)
  |
  |-- POS Adapter         -- Full staff UI, modifier selection, table mgmt
  |-- Waiter Adapter      -- Compact UI, quick-add, table assignment
  |-- QR Adapter          -- Guest-facing, simplified menu, payment focus
  |-- Kiosk Adapter       -- Large touch UI, visual menu, guided flow
  |-- Web Adapter         -- Browser-based, responsive, delivery address
  |-- Delivery Adapter    -- API integration, external order mapping
  |
  v
Order Engine (Core Domain)
  |
  |-- Ticket Creation     -- Create order with channel metadata
  |-- Line Item Management-- Add/remove/modify items, modifiers, combos
  |-- Price Calculation   -- Unit prices, modifiers, quantities
  |-- Discount Engine     -- Apply discounts (%, fixed, combo, loyalty)
  |-- Tax Calculation     -- VAT computation per country rules
  |-- Order Lifecycle     -- State machine: draft -> open -> preparing ->
  |                          ready -> served -> closed
  |-- Validation          -- Business rules (min order, item availability)
  |
  v
Downstream Consumers (Output Layer)
  |
  |-- Kitchen Router      -- Route to correct KDS/printer by item type
  |-- Payment Processor   -- Process payment regardless of channel
  |-- Fiscal Signer       -- TSE signing for all orders equally
  |-- Receipt Generator   -- Print/email/SMS receipt
  |-- Sync Engine         -- Sync to cloud and peer devices
  |-- Analytics           -- Track metrics per channel
```

### The Order Entity

Every order has a `channel` field that records its origin but does not
affect its processing:

```dart
enum OrderChannel {
  pos,        // Staff POS terminal
  waiter,     // Waiter handheld
  qrGuest,    // QR code guest order
  kiosk,      // Self-service kiosk
  web,        // Web ordering
  delivery,   // Third-party delivery platform
}

class Order {
  final String id;              // UUID v7
  final String branchId;
  final OrderChannel channel;   // Where the order came from
  final String? tableId;        // Null for takeaway/delivery
  final String? serverId;       // Staff who owns the order (null for guest)
  final OrderStatus status;
  final List<OrderLine> lines;
  final int subtotalCents;
  final int taxCents;
  final int totalCents;
  final DateTime createdAt;
  // ... payment references, fiscal data, etc.
}
```

### Channel Adapter Responsibilities

Each adapter is responsible for:

1. **Input capture**: Collecting order information in a channel-appropriate
   way (touch grid for POS, scrollable menu for QR, visual cards for kiosk).
2. **Validation**: Channel-specific validation (e.g., kiosk requires payment
   before order submission; POS allows open tabs).
3. **Transformation**: Converting channel-specific input into the common
   `OrderCommand` format that the Order Engine accepts.
4. **UX flow**: Managing the ordering flow appropriate for the channel
   (e.g., kiosk has a linear flow with cart review; POS allows non-linear
   editing).

### What Adapters Do NOT Do

Adapters never:
- Calculate prices, taxes, or discounts (Order Engine does this)
- Determine kitchen routing (Kitchen Router does this)
- Handle payment processing (Payment Processor does this)
- Generate receipts or fiscal data
- Make any business logic decisions beyond input transformation

### Kitchen Sees No Difference

The Kitchen Display System receives orders from the Order Engine's output
queue. A kitchen ticket looks identical whether it came from a POS terminal,
a QR code, or a kiosk:

```
┌─────────────────────────────┐
│ #A-0042        Table 7      │
│ Channel: QR     12:34 PM    │
│─────────────────────────────│
│ 1x Margherita Pizza         │
│   + Extra Mozzarella        │
│ 2x Caesar Salad             │
│ 1x Sparkling Water          │
│─────────────────────────────│
│ Notes: No croutons on salad │
└─────────────────────────────┘
```

The only visible difference is the `Channel` label, which helps kitchen
staff understand context (a QR order might need more preparation time
since the guest may not flag down a waiter for status).

## Consequences

### Positive

- **Single source of truth**: All business logic (pricing, tax, discounts,
  validation) exists in one place. A price change affects all channels
  immediately.
- **Consistency**: A Margherita Pizza costs the same whether ordered via
  POS, kiosk, or QR code. Tax calculation is identical. Fiscal signing
  is identical.
- **Easy to add channels**: Adding a new channel (e.g., voice ordering,
  drive-through) means building only a new adapter. The Order Engine,
  kitchen routing, payment, and fiscal signing work automatically.
- **Unified reporting**: Revenue per channel, items sold per channel,
  average order value per channel -- all from one orders table with a
  channel filter.
- **Simplified kitchen**: Kitchen staff learn one system. Orders queue
  in arrival order regardless of source.
- **Reduced testing**: Core business logic tested once. Each adapter
  tested for its input handling only.

### Negative

- **Adapter abstraction overhead**: Simple channels (POS) must go through
  the adapter abstraction even when they could talk to the Order Engine
  directly. This is minimal overhead but adds a layer of indirection.
- **Least-common-denominator risk**: The common Order model must accommodate
  all channels. Channel-specific features (e.g., delivery address for web,
  table assignment for waiter) are optional fields on the shared model.
- **Channel-specific pricing**: If a restaurant wants different prices for
  kiosk vs. dine-in, the Order Engine must support price overrides per
  channel. This adds complexity to the pricing logic.

### Risks

- **Model bloat**: As channels are added, the Order entity may accumulate
  optional fields specific to individual channels. Mitigation: use a
  `channel_metadata` JSON field for channel-specific data that does not
  affect core processing.
- **Channel-specific business rules**: Some channels may need different
  validation rules (e.g., kiosk requires prepayment, POS allows tabs).
  Mitigation: the Order Engine supports configurable validation policies
  per channel.
- **Third-party delivery integration complexity**: External platforms
  (Uber Eats, Lieferando) have their own order models, status flows,
  and APIs. The delivery adapter must handle significant translation
  complexity. Mitigation: delivery adapter is a Phase 6+ feature with
  dedicated development effort.

## Alternatives Considered

### Separate Order Systems per Channel

Each channel maintains its own order processing pipeline, with shared
reporting at the cloud level.

**Rejected because:**
- Business logic duplication across every channel (price calculation,
  tax, discounts, validation).
- Kitchen would need to monitor multiple order queues or a complex
  aggregation layer.
- Payment processing would be duplicated.
- A price change would need to be applied in multiple systems.
- Bug fixes in order logic would need to be replicated across channels.
- Testing effort multiplies with each channel.

### Channel-Specific Microservices

Each channel as a separate microservice that feeds into a shared kitchen
queue.

**Rejected because:**
- Microservices overhead (see ADR-001) for what is fundamentally the
  same business operation (creating an order).
- Distributed transaction complexity when a kiosk order needs payment
  + order creation + kitchen routing atomically.
- Operational complexity for a small team.

### Generic Order API Without Adapters

Single API endpoint that all channels call directly with no adapter layer.

**Rejected because:**
- Without adapters, channel-specific validation and transformation logic
  would be pushed to each client application, duplicating it across
  Flutter modules.
- Harder to enforce consistent input formatting.
- No clear place for channel-specific UX orchestration logic.
- Adapters provide a clean separation between "how the user interacts"
  and "how the system processes."
