# ADR-002: Local Branch Runtime Separate from Cloud

## Status
Accepted

## Date
2026-03-20

## Context

A restaurant is a high-pressure, real-time environment. When the lunch rush
hits, the POS system must work -- no exceptions. Internet outages, cloud
provider incidents, and DNS failures are not hypothetical scenarios; they are
regular occurrences, especially in locations with unreliable connectivity
(basement kitchens, rural areas, newly opened branches).

The core question is: **What happens when the internet goes down?**

In many competing POS platforms, the answer is "the system degrades or stops
working." This is unacceptable for our target market. A restaurant that cannot
take orders and payments because of an internet outage loses revenue and
customer trust.

We need an architecture where:
- The cloud being down never prevents a restaurant from operating.
- Taking orders, processing payments, printing receipts, and managing the
  kitchen display all work without any internet connectivity.
- Cloud connectivity is a welcome enhancement, not a hard dependency.

## Decision

We adopt a **Branch Runtime** architecture where each restaurant branch runs
a fully self-sufficient application tier on local devices.

### Branch Runtime (Offline-First)

The Branch Runtime is a Flutter application with an embedded SQLite database
that runs on Android tablets (primary) and web browsers (secondary). It is
the system of record during active service.

**The Branch Runtime can do the following with zero internet:**
- Create and manage orders (dine-in, takeaway, delivery intake)
- Apply discounts, modifiers, and combo logic
- Calculate taxes (cached tax tables)
- Process cash payments and track payment status
- Print kitchen tickets and receipts (via local network printers)
- Display orders on KDS (Kitchen Display System) via LAN
- Manage table assignments and floor plans
- Handle shift open/close and basic cash drawer reconciliation
- Authenticate staff via cached credentials (PIN-based)

**The Branch Runtime requires internet for:**
- Card payment processing (terminal communicates with acquirer)
- German fiscal signing via cloud TSE (Fiskaly) -- see ADR-008
- Syncing data to cloud for reporting and multi-branch management
- Receiving catalog updates pushed from cloud/ERPNext
- License validation (cached with grace period)

### Cloud Hub (Enhancement Layer)

The Cloud Hub (Go backend + PostgreSQL) provides:
- Multi-branch aggregation and reporting
- Centralized catalog and price management
- License and feature flag management
- Sync coordination between devices and branches
- Integration with ERPNext for accounting
- Fiscal compliance orchestration

The Cloud Hub is designed to be **eventually consistent** with Branch
Runtimes. It receives data when connectivity is available and never blocks
branch operations.

### Sync Model

```
Branch Runtime (Flutter/SQLite)
    |
    |-- LAN Sync (mDNS + WebSocket) --> Other devices in same branch
    |
    |-- Cloud Sync (HTTPS REST) --> Cloud Hub (Go/PostgreSQL)
                                        |
                                        +--> ERPNext Bridge
                                        +--> Fiskaly (Fiscal)
                                        +--> Reporting / Analytics
```

- **LAN Sync**: Real-time coordination between tablets in the same branch
  (table status, KDS updates). Sub-second latency. Works without internet.
- **Cloud Sync**: Batch sync of orders, payments, catalog changes. Runs
  when internet is available. Handles conflict resolution.

### Data Ownership

During active service, the Branch Runtime SQLite database is the **source
of truth** for that branch. The cloud stores a synchronized copy.

After end-of-day processing and successful sync, the cloud becomes the
**system of record** for historical data, reporting, and compliance.

## Consequences

### Positive

- **Uninterrupted service**: Internet outages do not impact restaurant
  operations. Staff may not even notice the cloud is down.
- **Low latency**: All POS interactions are local. No network round-trips
  for order creation, modification, or kitchen routing. UI feels instant.
- **Resilient deployment**: Each branch is independently operational. A
  cloud deployment issue affects reporting and sync, not revenue.
- **Reduced cloud costs**: The cloud does not need to handle real-time
  transaction processing for every order. It handles async sync and
  batch operations.
- **Simpler scaling**: Adding a new branch means deploying tablets with
  the app. No cloud scaling needed per-branch for real-time operations.

### Negative

- **Data duplication**: The same order exists in SQLite and PostgreSQL.
  Sync logic must handle conflicts, deduplication, and ordering.
- **Eventual consistency**: Reports in the cloud may lag behind actual
  branch state during connectivity gaps. Operators must understand this.
- **Two databases to maintain**: Schema evolution must be coordinated
  between Drift (SQLite) migrations and PostgreSQL migrations.
- **Offline limitations**: Some features inherently require internet
  (card payments, cloud TSE, real-time multi-branch dashboards).

### Risks

- **Sync conflicts**: Two devices modify the same order offline. Both
  sync later. Mitigation: conflict resolution strategy based on
  timestamps and operation type (last-writer-wins for most fields,
  merge for line items). See ADR-014 for sync details.
- **Data loss on device failure**: If a tablet dies before syncing,
  unsent transactions are lost. Mitigation: LAN sync between devices
  provides redundancy within the branch. Critical transactions (payments)
  trigger immediate sync attempts.
- **Stale cached data**: Catalog or price changes made in cloud may not
  reach an offline branch. Mitigation: branches pull catalog updates on
  every connectivity restoration. Critical price changes can be made
  directly on the lead device.

## Alternatives Considered

### Cloud-First with Offline Cache

Architecture where the cloud is the primary system and the local app
maintains a read-through cache for offline scenarios.

**Rejected because:**
- Treating offline as an exception rather than the default leads to
  brittle fallback logic and untested edge cases.
- Cache invalidation is one of the hardest problems in computer science.
  A POS system where cached data might be wrong is worse than one
  designed to work offline from the start.
- This architecture breaks under sustained offline periods (hours,
  not seconds).

### Progressive Web App (PWA) with Service Workers

Browser-based PWA with service worker caching for offline support.

**Rejected because:**
- Service worker storage limits (browser-dependent, often 50-200MB)
  are insufficient for a full branch dataset.
- Browser-based offline is fragile -- clearing cache, browser updates,
  or storage pressure can wipe the offline data.
- No direct access to hardware (receipt printers, cash drawers,
  payment terminals) without additional bridge software.
- Cannot guarantee the same level of reliability as a native app
  with SQLite.

### Thick Client with Embedded Server

Traditional thick-client architecture with a local server process
running on a branch server machine.

**Rejected because:**
- Requires a dedicated server at each branch (cost, maintenance,
  hardware failures).
- Adds operational complexity -- restaurants are not IT departments.
- Flutter on Android tablets provides sufficient compute power for
  a single-branch POS workload without a separate server.
- The lead tablet can serve as the branch coordinator via LAN sync.
