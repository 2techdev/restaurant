# ADR-008: Cloud TSE Fiscal Finalization for Germany

## Status
Accepted

## Date
2026-03-20

## Context

### German Fiscal Law (KassenSichV)

Since January 1, 2020, German law requires all electronic cash register
systems (Kassensysteme) to use a **Technical Security Equipment** (Technische
Sicherheitseinrichtung, TSE) for tamper-proof recording of transactions.

Key requirements:

- **Every transaction** (Kassenvorgang) must be signed by a certified TSE.
- The TSE produces a cryptographic signature, a transaction counter, and a
  serial number that must appear on the receipt.
- **DSFinV-K** (Digitale Schnittstelle der Finanzverwaltung fuer
  Kassensysteme) defines the export format for fiscal audits.
- The Finanzamt (tax authority) can audit any restaurant at any time and
  expects DSFinV-K compliant data exports.
- Non-compliance can result in fines and tax estimates.

### TSE Options

Two categories of TSE exist:

1. **Hardware TSE**: Physical device (USB stick, SD card, or network-attached)
   connected to the POS. Examples: Swissbit, Epson, Diebold Nixdorf.
   - Pros: Works offline, no internet required
   - Cons: Hardware procurement, per-device cost, firmware updates,
     physical failures, device lock-in

2. **Cloud TSE**: Cloud-based service that provides TSE functionality via API.
   Examples: Fiskaly SIGN DE, Deutsche Fiskal.
   - Pros: No hardware, instant provisioning, centralized management,
     vendor handles compliance updates
   - Cons: Requires internet connectivity for signing

### The Tension

Our architecture is offline-first (ADR-002). The Branch Runtime operates
independently of the internet. However, German fiscal law requires TSE
signing of every transaction. Cloud TSE requires internet. This creates
a fundamental tension.

## Decision

We adopt **Fiskaly SIGN DE v2** (cloud TSE) as the primary fiscal signing
solution for Germany, with a defined offline-tolerance pattern and a planned
hardware TSE fallback for Phase 8+.

### Architecture

```
Branch Runtime (Flutter/SQLite)
    |
    | 1. Order completed, payment recorded locally
    | 2. Transaction queued for fiscal signing
    |
    v
Cloud Hub (Go/PostgreSQL)
    |
    | 3. Receives transaction via sync
    | 4. Calls Fiskaly SIGN DE v2 API
    |
    v
Fiskaly Cloud TSE
    |
    | 5. Returns signed receipt data:
    |    - Signature
    |    - Transaction number
    |    - TSE serial number
    |    - Start/finish timestamps
    |
    v
Cloud Hub
    |
    | 6. Stores fiscal data
    | 7. Syncs signed receipt back to device
    |
    v
Branch Runtime
    |
    | 8. Prints receipt with TSE data
    | 9. Can reprint with QR code containing fiscal info
```

### Offline Transaction Flow

During normal operation (internet available):

1. Order completed on tablet
2. Sync engine sends to Cloud Hub immediately
3. Cloud Hub calls Fiskaly within seconds
4. Signed receipt data syncs back to tablet
5. Receipt printed with full TSE information

During internet outage:

1. Order completed on tablet
2. Transaction stored locally with `fiscal_status = PENDING`
3. Receipt printed with note: "Fiscalization pending - see final receipt"
4. When internet returns, sync engine sends backlog to Cloud Hub
5. Cloud Hub processes Fiskaly signing for all pending transactions
6. Signed data syncs back; `fiscal_status = SIGNED`
7. Final receipts available for reprint

### Fiskaly Integration Details

```
API Base: https://sign.fiskaly.com/api/v2

Key Operations:
- PUT /tss/{tss_id}/tx/{tx_id}?tx_revision=1  -- Start transaction
- PUT /tss/{tss_id}/tx/{tx_id}?tx_revision=2  -- Finish transaction

Per-Tenant Resources:
- TSS (Technical Security System) -- one per branch
- Client -- one per POS device
```

Each tenant (restaurant) gets:
- One Fiskaly TSS per branch
- One Fiskaly client per registered POS device
- API key and secret stored encrypted in Cloud Hub

### DSFinV-K Export

The Cloud Hub generates DSFinV-K compliant exports from its PostgreSQL
database:

- `transactions.csv` -- All signed transactions
- `lines.csv` -- Line items per transaction
- `payment_types.csv` -- Payment method breakdown
- `cashpointclosing.csv` -- Z-reports (shift/day closings)
- `slaves.csv` -- Registered POS devices

Export is generated on demand (Finanzamt audit) or scheduled (monthly).

## Consequences

### Positive

- **No hardware dependency**: No USB TSE to procure, ship, install, and
  maintain at each branch. Restaurants can go live faster.
- **Centralized management**: All TSE operations visible in Cloud Hub
  dashboard. Compliance status per branch is transparent.
- **Automatic updates**: Fiskaly handles updates when regulations change.
  No firmware updates to push to hundreds of devices.
- **Scalable provisioning**: Adding a new branch or device is an API call,
  not a hardware shipment.
- **Simpler development**: REST API integration is straightforward compared
  to platform-specific USB/serial interfaces for hardware TSE.
- **Audit readiness**: All fiscal data centralized in PostgreSQL makes
  DSFinV-K export generation straightforward.

### Negative

- **Internet dependency for fiscal compliance**: Transactions cannot be
  fiscally signed during internet outages. The restaurant can still
  operate (orders, payments work offline), but receipts lack TSE data
  until connectivity returns.
- **Ongoing cost**: Fiskaly charges per TSS per month (approximately
  30-40 EUR/month per branch). Hardware TSE is a one-time purchase
  (approximately 100-200 EUR per device).
- **Vendor dependency**: We depend on Fiskaly's uptime and API stability.
  If Fiskaly has an outage, no transactions can be signed until recovery.
- **Latency**: Each signing request adds network round-trip time
  (approximately 100-500ms). Batch processing mitigates this.

### Risks

- **Fiskaly service outage**: If Fiskaly is down, fiscal signing stalls.
  Mitigation: implement retry with exponential backoff, alert on
  prolonged failures, document offline tolerance window for compliance.
- **Regulatory interpretation**: Some Finanzamt auditors may question the
  delay between transaction and TSE signing during offline periods.
  Mitigation: the system logs exact timestamps for transaction creation
  and TSE signing. The delay is documented and justified by the
  offline-first architecture. Fiskaly's cloud TSE is BSI-certified,
  which provides regulatory cover.
- **Internet requirement in rural areas**: Some German restaurants may
  have unreliable internet. Extended offline periods could accumulate
  large unsigned transaction backlogs. Mitigation: hardware TSE fallback
  planned for Phase 8+.
- **Cost escalation**: As we add branches, Fiskaly costs grow linearly.
  Mitigation: negotiate volume pricing, evaluate cost/benefit of
  hardware TSE for high-volume branches.

## Alternatives Considered

### Hardware TSE (Swissbit USB / Epson network TSE)

Physical TSE device connected directly to the POS tablet or branch
network.

**Not rejected permanently** -- planned as a Phase 8+ fallback option.

**Deferred because:**
- Requires Android USB host mode support and platform-specific native
  code for each TSE manufacturer.
- Hardware procurement and logistics for initial rollout slow down
  time-to-market.
- Each manufacturer has a different SDK/API, increasing integration
  complexity.
- Firmware update management across deployed devices is operationally
  complex.

**Will be implemented when:**
- Customers require guaranteed offline fiscal signing.
- Rural deployments with unreliable internet become a significant
  segment.
- Cost optimization makes hardware TSE attractive for high-volume
  branches.

### Deutsche Fiskal Cloud TSE

Alternative cloud TSE provider to Fiskaly.

**Evaluated but not chosen because:**
- Fiskaly has wider adoption in the German POS market and more
  comprehensive documentation.
- Fiskaly's API is simpler and better documented.
- Deutsche Fiskal was considered as a potential backup provider.
- The bridge module is designed to support multiple TSE providers
  via a common interface (`FiscalProvider`), so switching is possible.

### Self-Hosted TSE

Running our own TSE-certified signing infrastructure.

**Rejected because:**
- TSE certification by BSI (Bundesamt fuer Sicherheit in der
  Informationstechnik) requires extensive security audits and
  compliance processes.
- Maintaining certification requires ongoing compliance work.
- This is not our core competency and would distract from POS
  development.
- The certification process alone can take 12-18 months.

### No Cloud TSE -- Hardware Only

Skip cloud TSE entirely and use only hardware TSE.

**Rejected because:**
- Hardware TSE adds significant time-to-market delay for the German
  launch.
- Initial hardware costs and logistics complexity for first customers.
- Cloud TSE allows us to launch in Germany faster and add hardware
  TSE later as an alternative.
