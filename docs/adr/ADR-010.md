# ADR-010: Immutable Transaction Log with Mutable State

## Status
Accepted

## Date
2026-03-20

## Context

A restaurant POS system deals with two fundamentally different types of data:

### Transactional Data (Financial Records)

Orders, payments, receipts, voids, refunds, shift closings, cash drawer
movements. This data has **legal and fiscal significance**:

- **Germany (GoBD)**: Grundsaetze zur ordnungsmaessigen Fuehrung und
  Aufbewahrung von Buechern requires that all financially relevant data
  be stored in an unalterable form. Deletions and modifications of
  completed transactions are prohibited.
- **Germany (KassenSichV)**: TSE-signed transactions must be retained
  for 10 years and must be reproducible for audit.
- **Switzerland**: VAT records must be retained for 10 years.
- **General accounting**: Audit trails require knowing exactly what
  happened, when, and by whom.

### Master Data (Configuration)

Products, categories, prices, users, floor plans, printer configurations,
tax rates. This data **changes over time** as the business evolves:

- Menu items are added, renamed, or discontinued
- Prices change seasonally or due to cost increases
- Staff join and leave
- Floor plans are rearranged

The challenge is designing a data model that:
1. Never allows modification or deletion of financial records
2. Supports natural evolution of master data
3. Works with offline sync (ADR-002, ADR-014)
4. Is auditable and compliant across multiple jurisdictions

## Decision

We adopt a **dual model**: an immutable append-only transaction log for
all financial data, and mutable records with version tracking for master
data.

### Immutable Transaction Log

All financial events are recorded as **immutable, append-only** records:

```
┌─────────────────────────────────────────────────────┐
│ Transaction Log (append-only)                       │
│─────────────────────────────────────────────────────│
│ OrderCreated       { order_id, lines, totals, ... } │
│ PaymentReceived    { payment_id, order_id, amount }  │
│ ReceiptIssued      { receipt_id, tse_signature }     │
│ OrderVoided        { void_id, original_order_id }    │
│ RefundIssued       { refund_id, original_payment_id }│
│ ShiftClosed        { shift_id, cash_totals }         │
│ CashDrawerMovement { amount, reason, operator }      │
└─────────────────────────────────────────────────────┘
```

**Rules for the transaction log:**

1. **No UPDATE**: Once a record is written, it is never modified. Not by
   application code, not by migration scripts, not by admin tools.
2. **No DELETE**: Records are never removed. Not even soft-deleted.
   Fiscal transaction records exist forever.
3. **Voids are new records**: Voiding an order does not modify the
   original order record. Instead, a new `OrderVoid` record is created
   that references the original order. The original order remains in
   its completed state with a cross-reference to the void.
4. **Refunds are new records**: Similarly, refunding a payment creates
   a new `Refund` record referencing the original payment.
5. **Corrections are new records**: If an order total was calculated
   incorrectly, a corrective record is appended, never an UPDATE to
   the original.

### Order State Machine

```
draft --> open --> preparing --> ready --> served --> closed
                                                       |
                                                       +--> voided (new record)
```

Each state transition is an immutable event. The current state is derived
from the latest event for that order.

### Mutable Master Data with Version Tracking

Master data records are mutable but tracked:

```sql
CREATE TABLE products (
    id          UUID PRIMARY KEY,
    name        TEXT NOT NULL,
    price_cents INTEGER NOT NULL,
    category_id UUID REFERENCES categories(id),
    tax_rate_id UUID REFERENCES tax_rates(id),
    is_active   BOOLEAN NOT NULL DEFAULT true,
    is_deleted  BOOLEAN NOT NULL DEFAULT false,
    version     INTEGER NOT NULL DEFAULT 1,
    updated_at  TIMESTAMP NOT NULL,
    updated_by  UUID REFERENCES users(id),
    created_at  TIMESTAMP NOT NULL
);
```

**Rules for master data:**

1. **Mutable**: Fields can be updated (name, price, active status).
2. **Version tracked**: Each update increments the `version` field.
   The sync engine uses versions to detect and resolve conflicts.
3. **Soft delete only**: Products are never physically deleted (see
   ADR-015). `is_deleted = true` hides them from active use.
4. **Snapshot on transaction**: When an order line is created, the
   product name and price at that moment are **copied** into the order
   line record. This ensures that future price changes do not alter
   historical order data.

### Snapshot Pattern for Transactional Integrity

```dart
class OrderLine {
  final String id;
  final String orderId;
  final String productId;       // Reference to current product
  final String productName;     // Snapshot: name at order time
  final int unitPriceCents;     // Snapshot: price at order time
  final String taxRateCode;     // Snapshot: tax rate at order time
  final int taxRateValue;       // Snapshot: tax percentage at order time
  final int quantity;
  final int lineTotalCents;     // Calculated from snapshot values
}
```

When a product's price changes from 12.50 to 13.00, all future orders
use 13.00. All past orders retain 12.50 in their line items. The
historical order total remains correct and auditable.

## Consequences

### Positive

- **Legal compliance**: GoBD, KassenSichV, and Swiss VAT requirements
  are satisfied by design. Financial records cannot be tampered with.
- **Complete audit trail**: Every financial event is recorded with
  timestamp, operator, and full context. Auditors can reconstruct the
  exact state of any transaction at any point in time.
- **Sync reliability**: Immutable records are easier to sync. An
  append-only log never has conflicting updates -- only new records
  to propagate.
- **Historical accuracy**: Price changes, menu changes, and tax rate
  changes never retroactively affect past transactions. Reports are
  accurate for any historical period.
- **Forensic capability**: In case of disputes (customer claims they
  were overcharged, staff suspected of fraud), the full transaction
  history is available and untampered.

### Negative

- **Storage growth**: Data is never deleted, so storage grows
  monotonically. Mitigation: POS transaction data is relatively small
  (kilobytes per order). A busy restaurant generating 500 orders/day
  accumulates roughly 50-100MB per year of transaction data. Modern
  storage costs make this negligible.
- **Void complexity**: Voiding an order requires creating a new void
  record, updating the materialized state, and ensuring the void
  appears correctly in reports and fiscal exports. More complex than
  simply deleting the order.
- **Query complexity**: To determine an order's current state, the
  system must consider the base order record plus any void/refund
  records. Mitigation: maintain a materialized `current_status` field
  on the order record (updated by the void/refund process) for fast
  reads, while the immutable log provides the authoritative history.
- **Migration constraints**: Schema migrations on immutable tables
  are limited to additive changes (new columns with defaults). Cannot
  rename or remove columns from transaction tables.

### Risks

- **Accidental mutation**: A developer might write an UPDATE query
  against a transaction table. Mitigation: database-level triggers
  or row-level security policies that prevent UPDATE/DELETE on
  transaction tables. Code review checks. CI tests that verify
  no migration contains UPDATE/DELETE on transaction tables.
- **Snapshot staleness**: If a product changes during order creation
  (between adding items and submitting), the snapshot values might
  be inconsistent. Mitigation: snapshot is taken at line item
  addition time. If the product changes while the order is still
  being built, the UI notifies the user.
- **Storage on mobile devices**: SQLite on tablets has limited
  storage compared to PostgreSQL on cloud servers. Mitigation:
  archival process moves completed orders older than N months to
  a compressed format. Cloud retains full history. Devices retain
  recent history.

## Alternatives Considered

### Event Sourcing

Store only events, derive all state from replaying events.

**Not fully adopted because:**
- Full event sourcing adds significant complexity (event store,
  projections, snapshots for read performance, eventual consistency).
- For a POS system, the current state is queried far more often
  than the event history. Materializing state from events on every
  read is too expensive.
- We adopt the core principle (immutable events for financial data)
  without the full event sourcing infrastructure.
- Hybrid approach: immutable transaction log + materialized current
  state gives us the benefits of both.

### Mutable Everything with Audit Log

Allow updates to all records but log every change in a separate
audit table.

**Rejected because:**
- Audit logs can be tampered with if they are not the primary
  record. The "real" data is the mutable record; the audit log
  is a secondary copy.
- Complexity of maintaining a shadow audit table for every
  entity.
- Does not satisfy GoBD requirements which mandate that the
  primary record itself be unalterable.

### Full Immutability (Including Master Data)

Make everything immutable, including products and configuration.

**Rejected because:**
- Impractical for daily operations. Changing a product name would
  require creating a new product record and re-linking all
  references.
- Overly complex for data that has no legal immutability requirement.
- Version tracking on mutable master data provides sufficient
  audit capability for non-financial records.
