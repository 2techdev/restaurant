# ADR-006: SQLite for Local Runtime Database

## Status
Accepted

## Date
2026-03-20

## Context

The Branch Runtime (Flutter application on Android tablets) requires a local
database that supports offline-first operation. Every order, payment, product,
and configuration record must be available locally without any network
connectivity.

Requirements for the local database:

1. **Embedded**: Must run inside the Flutter application process. No separate
   database server to install or manage.
2. **Relational**: POS data is inherently relational -- orders have line items,
   line items reference products, products belong to categories, payments
   reference orders. A relational model with foreign keys and joins is the
   natural fit.
3. **Reliable**: Data loss is unacceptable in a financial system. The database
   must handle crashes, power failures, and unexpected shutdowns gracefully.
4. **Performant**: Must support fast writes during peak hours (hundreds of
   order modifications per hour) and fast reads for UI reactivity.
5. **Migration support**: Schema will evolve over months and years. Each
   tablet in the field must be upgradeable from any previous schema version.
6. **Encryption**: Local data on a restaurant tablet must be encrypted at
   rest to protect against device theft.
7. **Reactive**: UI should update automatically when underlying data changes
   (e.g., KDS display updates when a new order is created on another device
   and synced locally).

## Decision

We adopt **SQLite** as the local database engine, accessed through the
**Drift ORM** (formerly Moor) for Flutter/Dart.

### Why SQLite

SQLite is the most widely deployed database engine in the world. It runs on
every Android device, is tested with billions of deployments, and has a
well-defined transaction model with ACID guarantees.

Key SQLite configurations for our use case:

```sql
PRAGMA journal_mode = WAL;          -- Write-Ahead Logging for concurrent reads
PRAGMA synchronous = NORMAL;        -- Balance between safety and performance
PRAGMA foreign_keys = ON;           -- Enforce referential integrity
PRAGMA busy_timeout = 5000;         -- Wait up to 5s for locks
PRAGMA cache_size = -8000;          -- 8MB page cache
```

### Why Drift ORM

Drift is a reactive, type-safe SQLite library for Dart/Flutter maintained by
Simon Binder. It provides:

- **Code generation**: Dart classes generated from table definitions. No
  hand-written mapping code.
- **Type safety**: Queries are checked at compile time. Column types,
  nullability, and foreign keys are enforced in Dart.
- **Reactive streams**: `watch()` queries return `Stream<List<T>>` that
  emit new results whenever the underlying tables change. This drives
  the reactive UI pattern throughout the app.
- **Migration framework**: Versioned migrations with `onUpgrade` callbacks.
  Each schema change is a numbered migration. Drift can generate migration
  steps between any two versions.
- **Raw SQL escape hatch**: When the query builder is insufficient, raw
  SQL queries are fully supported with type-safe result mapping.

### Schema Design (Simplified)

```dart
class Orders extends Table {
  TextColumn get id => text()();            // UUID v7
  TextColumn get branchId => text()();
  TextColumn get channel => text()();       // pos, kiosk, qr, web
  IntColumn get status => intEnum<OrderStatus>()();
  IntColumn get subtotalCents => integer()();
  IntColumn get taxCents => integer()();
  IntColumn get totalCents => integer()();
  DateTimeColumn get createdAt => dateTime()();
  DateTimeColumn get updatedAt => dateTime()();
  TextColumn get createdBy => text()();
  BoolColumn get isDeleted => boolean().withDefault(const Constant(false))();
  IntColumn get syncVersion => integer().withDefault(const Constant(0))();

  @override
  Set<Column> get primaryKey => {id};
}

class OrderLines extends Table {
  TextColumn get id => text()();
  TextColumn get orderId => text().references(Orders, #id)();
  TextColumn get productId => text()();
  TextColumn get productName => text()();   // Snapshot at order time
  IntColumn get quantity => integer()();
  IntColumn get unitPriceCents => integer()();
  IntColumn get lineTotalCents => integer()();
  // ... modifiers, discounts, tax rate
}
```

### Encryption with SQLCipher

For data-at-rest encryption, we use **SQLCipher** (via the
`sqlcipher_flutter_libs` package):

- AES-256 encryption of the entire database file
- Encryption key derived from a device-specific secret stored in
  Android Keystore
- Transparent to Drift -- all queries work identically
- Performance overhead: approximately 5-15% slower than unencrypted
  SQLite (negligible for POS workload)

### Reactive UI Pattern

```dart
// In a Riverpod provider
Stream<List<OrderWithLines>> watchActiveOrders(String branchId) {
  return (select(orders)
    ..where((o) => o.branchId.equals(branchId))
    ..where((o) => o.status.isIn([
      OrderStatus.open.index,
      OrderStatus.inProgress.index,
    ]))
    ..orderBy([(o) => OrderingTerm.asc(o.createdAt)])
  ).join([
    leftOuterJoin(orderLines, orderLines.orderId.equalsExp(orders.id)),
  ]).watch().map(groupByOrder);
}
```

The KDS screen, POS order list, and table floor plan all use `watch()`
queries. When a sync operation inserts or updates records, all watching
UI components rebuild automatically.

## Consequences

### Positive

- **Battle-tested reliability**: SQLite has been deployed on billions of
  devices. Its crash recovery, WAL mode, and ACID compliance are proven
  beyond any alternative.
- **Zero administration**: No database server, no connection strings, no
  authentication. The database is a file on the device.
- **Drift ORM excellence**: Type-safe queries, compile-time validation,
  reactive streams, and excellent migration support make development
  fast and safe.
- **Familiar SQL**: Team members can write and debug queries using
  standard SQL. No proprietary query language to learn.
- **Lightweight**: SQLite adds approximately 1MB to the application size.
  Database files for a typical branch (thousands of orders) remain under
  100MB.
- **Concurrent reads**: WAL mode allows multiple read transactions
  concurrent with one write transaction. The UI thread can read while
  the sync engine writes.

### Negative

- **Single-writer limitation**: SQLite allows only one write transaction
  at a time. Concurrent writes from UI thread and sync engine must be
  serialized. Mitigation: Drift handles this internally with a
  sequential write queue.
- **No built-in replication**: SQLite has no native replication. Our
  sync engine must implement replication logic manually (see ADR-014).
- **Schema divergence risk**: SQLite migrations on devices and PostgreSQL
  migrations on the cloud must be kept in sync conceptually, but are
  maintained as separate migration sets.
- **Limited query optimizer**: SQLite's query planner is simpler than
  PostgreSQL's. Complex analytical queries may need careful index
  planning.

### Risks

- **Migration failures on devices**: A failed migration on a deployed
  tablet could brick the local database. Mitigation: Drift supports
  transactional migrations (rollback on failure). We also implement
  pre-migration backup and recovery.
- **Database corruption**: Rare but possible on Android devices with
  aggressive battery optimization or sudden power loss. Mitigation:
  WAL mode and `PRAGMA integrity_check` on app startup. Corrupted
  databases can be rebuilt from cloud sync.
- **Storage limits**: Low-end Android tablets may have limited storage.
  Mitigation: archival process moves old completed orders to a
  compressed archive table. Active database stays compact.

## Alternatives Considered

### Hive (Flutter NoSQL)

Key-value/document store for Flutter, pure Dart implementation.

**Rejected because:**
- NoSQL model is a poor fit for relational POS data. Joins, foreign
  keys, and aggregation queries are fundamental to POS operations.
- No SQL query language -- all queries are programmatic.
- Limited migration support for schema evolution.
- Less battle-tested than SQLite for data integrity.
- Hive v2 (rewrite) had an uncertain future at the time of evaluation.

### Isar

Modern NoSQL database for Flutter by the Hive author.

**Rejected because:**
- Same relational model concerns as Hive.
- Newer project with less production track record.
- Query language is proprietary, not SQL.
- Migration tooling is less mature than Drift.

### ObjectBox

High-performance object database for Flutter.

**Rejected because:**
- Proprietary license for commercial use.
- NoSQL model does not fit relational POS data naturally.
- Vendor lock-in risk with a single-vendor database.

### Room (Android) / Core Data (iOS)

Platform-native SQLite wrappers.

**Rejected because:**
- Room is Kotlin/Java only -- not available in Flutter/Dart.
- Core Data is iOS/macOS only.
- Using platform-specific database wrappers defeats the purpose of
  cross-platform Flutter development.
- Drift provides equivalent (and arguably superior) functionality
  in pure Dart.

### Realm (MongoDB)

Cross-platform embedded database with built-in sync.

**Rejected because:**
- Proprietary sync tied to MongoDB Atlas. We need custom sync logic
  for our Cloud Hub architecture.
- License concerns for commercial POS use.
- Object model (not relational) is a poor fit.
- Closed-source sync protocol limits our control.
