# ADR-005: Go for Cloud Backend

## Status
Accepted

## Date
2026-03-20

## Context

The Cloud Hub requires a backend language and runtime that serves multiple
purposes:

- **REST API server**: Handle sync requests from Branch Runtimes, serve the
  web dashboard, expose management APIs.
- **Background workers**: Process end-of-day summaries, run ERPNext bridge
  sync, generate reports, handle fiscal compliance workflows.
- **Integration layer**: Communicate with external services (Fiskaly, payment
  processors, ERPNext, push notification services).
- **Multi-tenant**: Serve multiple restaurant chains, each with multiple
  branches, from a single deployment.

Requirements for the backend language:

1. **Simple deployment**: We deploy to VPS instances and potentially
   self-hosted customer environments. The deployment artifact should be as
   simple as possible.
2. **Performance**: Handle concurrent sync requests from hundreds of devices
   across multiple branches without excessive resource consumption.
3. **Maintainability**: A small team (1-5 devs) must be able to understand,
   debug, and modify the entire codebase. The language should encourage
   readable, straightforward code.
4. **Ecosystem**: Sufficient library support for PostgreSQL, HTTP, JSON,
   WebSocket, cryptography, and standard web development tasks.
5. **Strong typing**: Financial calculations and multi-tenant data handling
   require compile-time safety.

## Decision

We adopt **Go 1.22+** for the Cloud Hub backend.

### Key Design Principles

- **Standard library first**: Use Go's standard library (`net/http`,
  `encoding/json`, `database/sql`, `crypto`, `context`) wherever possible.
  Add dependencies only when the standard library is genuinely insufficient.
- **Minimal dependencies**: Each added dependency is a maintenance liability.
  Prefer well-established, focused libraries over large frameworks.
- **Single binary**: The Cloud Hub compiles to a single statically-linked
  binary with no runtime dependencies (no JVM, no Node.js, no interpreter).

### Dependency Choices

| Concern              | Choice                | Rationale                          |
|----------------------|-----------------------|------------------------------------|
| HTTP routing         | `net/http` (Go 1.22+) | Built-in path params since 1.22   |
| Database             | `pgx` (jackc/pgx)    | Pure Go PostgreSQL driver, fastest |
| Migrations           | `goose`               | SQL-based, no ORM magic            |
| Configuration        | `envconfig` or `env`  | 12-factor app, env vars            |
| Logging              | `slog` (std lib)      | Structured logging, built-in       |
| Testing              | `testing` (std lib)   | Standard, with testify for asserts |
| JSON                 | `encoding/json`       | Standard library                   |
| UUID                 | `google/uuid`         | UUID v7 support                    |
| Middleware           | Custom                | Simple function wrapping           |

### HTTP API Design

```go
mux := http.NewServeMux()

// Go 1.22+ method-aware routing
mux.HandleFunc("POST /api/v1/sync", syncHandler)
mux.HandleFunc("GET /api/v1/orders/{id}", getOrderHandler)
mux.HandleFunc("POST /api/v1/orders", createOrderHandler)

// Middleware chain
handler := middleware.Chain(
    middleware.RequestID,
    middleware.Logging,
    middleware.TenantContext,
    middleware.Auth,
)(mux)
```

### Concurrency Model

Go's goroutines and channels are well-suited for the Cloud Hub's workload:

- Each HTTP request runs in its own goroutine (automatic with `net/http`).
- Background workers (ERPNext sync, report generation) run as long-lived
  goroutines managed by `context.Context` for graceful shutdown.
- Database connection pooling handled by `pgx` pool.
- No need for external task queues initially -- Go's concurrency
  primitives handle the current scale.

## Consequences

### Positive

- **Single binary deployment**: `GOOS=linux GOARCH=amd64 go build` produces
  one file. Copy to server, run it. No runtime installation, no package
  manager, no dependency resolution on the target machine.
- **Low resource usage**: Go binaries typically use 10-50MB of RAM for a
  web server handling moderate load. This matters for self-hosted deployments
  on modest VPS instances.
- **Fast compilation**: Full project compiles in seconds. Fast CI/CD cycles.
- **Built-in concurrency**: Goroutines handle thousands of concurrent sync
  requests without callback hell or thread pool configuration.
- **Readable code**: Go's simplicity (no generics abuse, no inheritance,
  explicit error handling) means any team member can read any module.
- **Excellent PostgreSQL support**: `pgx` is the best PostgreSQL driver in
  any language ecosystem -- pure Go, supports all PG features, connection
  pooling, prepared statements, COPY protocol.
- **Strong standard library**: HTTP server, JSON, crypto, testing, logging
  -- all built-in and production-quality.

### Negative

- **Verbose error handling**: `if err != nil { return err }` on every
  operation. No exceptions, no try-catch. This is intentional in Go's design
  but adds line count.
- **Limited generics**: Go 1.22 has generics but the ecosystem has not fully
  adopted them. Some utility code requires more boilerplate than in Rust or
  TypeScript.
- **No ORM**: We chose not to use an ORM (like GORM) in favor of raw SQL
  with `pgx`. This means writing SQL by hand, but gives full control over
  query performance.
- **Smaller web framework ecosystem**: Compared to Node.js/Express or
  Python/Django, Go has fewer batteries-included web frameworks. We build
  more infrastructure ourselves.

### Risks

- **Hiring pool**: Fewer Go developers than JavaScript or Python developers.
  Mitigation: Go is easy to learn for developers coming from any C-family
  language. The language specification fits in a short document.
- **Ecosystem churn**: Some Go libraries are abandoned by maintainers.
  Mitigation: prefer standard library and well-established libraries
  (pgx, goose) with active maintenance.
- **Over-engineering concurrency**: Go makes it easy to spawn goroutines,
  but also easy to create race conditions. Mitigation: use `go vet`,
  `race detector`, and structured concurrency patterns (errgroup).

## Alternatives Considered

### Node.js (TypeScript)

Server-side JavaScript with TypeScript for type safety.

**Rejected because:**
- Deployment requires Node.js runtime and `node_modules` directory
  (often hundreds of megabytes of dependencies).
- Single-threaded event loop requires clustering for CPU-bound work.
  Go's goroutines are simpler for concurrent workloads.
- NPM dependency tree is a security and maintenance concern (left-pad
  incidents, supply chain attacks).
- TypeScript's type system is unsound (any, type assertions) compared
  to Go's compile-time guarantees.

### Rust

Systems programming language with memory safety and high performance.

**Rejected because:**
- Steep learning curve (borrow checker, lifetimes, trait system).
  Development speed would be significantly slower for a small team.
- Longer compilation times compared to Go.
- Overkill for a web API server -- Go's performance is more than
  sufficient for our workload (we are not building a database or
  game engine).
- Smaller web ecosystem. Async Rust (tokio) adds complexity.

### Python (Django / FastAPI)

Python with Django for batteries-included web development or FastAPI
for modern async APIs.

**Rejected because:**
- GIL (Global Interpreter Lock) limits true concurrency. Workarounds
  (multiprocessing, async) add complexity.
- Deployment requires Python runtime, virtual environment, and
  dependency management (pip/poetry).
- Runtime type checking (even with type hints) catches fewer errors
  at compile time compared to Go.
- Performance: Python is 10-50x slower than Go for CPU-bound tasks.
  While most POS workload is I/O-bound, report generation and batch
  sync processing benefit from Go's speed.

### Java (Spring Boot) / Kotlin (Ktor)

JVM-based backend with Spring or Ktor framework.

**Rejected because:**
- JVM startup time and memory footprint (hundreds of MB minimum)
  are excessive for small deployments.
- Spring Boot's magic (dependency injection, auto-configuration,
  annotations) makes debugging harder for a small team.
- Deployment requires JVM installation and configuration.
- Go's simplicity and single-binary deployment are more aligned
  with our operational constraints.
