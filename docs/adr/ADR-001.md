# ADR-001: Modular Monolith Over Microservices

## Status
Accepted

## Date
2026-03-20

## Context

The restaurant POS platform requires a cloud backend (Cloud Hub) to handle
multi-branch management, reporting, sync coordination, license management, and
integration with third-party services (ERPNext, Fiskaly, payment gateways).

The founding team consists of 1-5 developers. We need to ship a production-ready
MVP within months, not years. At the same time, the platform must be structured
well enough to scale organizationally -- as the team grows, different developers
should be able to own different functional areas without stepping on each other.

The key tension is: **ship fast now** vs. **maintain clean boundaries for the
future**.

Three architectural patterns were evaluated:

1. **Microservices** -- each domain (orders, payments, tenants, sync, fiscal)
   as an independently deployed service with its own database.
2. **Traditional monolith** -- single codebase with no internal structure,
   shared database, everything intertwined.
3. **Modular monolith** -- single deployable binary with explicit internal
   module boundaries, shared database, communication via well-defined
   interfaces.

## Decision

We adopt a **modular monolith** architecture for the Cloud Hub.

The Go backend is structured as a single deployable binary with the following
characteristics:

- **Internal module boundaries**: Each domain (orders, payments, tenants, sync,
  fiscal, catalog, reporting) is a separate Go package with a clearly defined
  public API (exported types and functions).
- **Shared PostgreSQL database**: All modules share one PostgreSQL 16 instance
  but own their own tables. Cross-module queries go through the owning module's
  API, never by direct table access.
- **Dependency direction**: Modules depend on shared kernel types (money, IDs,
  time) and on each other's interfaces, never on concrete implementations.
- **Inter-module communication**: In-process function calls through interfaces.
  No HTTP calls between modules. No message queues between modules (yet).
- **Single deployment unit**: One `docker run` or one binary on the server.
  One health check endpoint. One log stream.

### Module Boundaries

```
cloud-hub/
  internal/
    tenant/       -- Multi-tenancy, branch management
    catalog/      -- Products, categories, modifiers, pricing
    order/        -- Order processing, ticket lifecycle
    payment/      -- Payment methods, reconciliation
    fiscal/       -- Country-specific fiscal compliance (TSE, DSFinV-K)
    sync/         -- Device sync coordination, conflict resolution
    auth/         -- Authentication, authorization, API keys
    license/      -- License management, feature flags
    reporting/    -- Analytics, exports, dashboards
    notification/ -- Push notifications, alerts
  pkg/
    money/        -- Money type (integer cents)
    identity/     -- UUID v7 generation and validation
    clock/        -- Time abstraction for testing
    errors/       -- Structured error types
```

### Rules

1. A module exposes a `Service` interface. Other modules depend on the interface.
2. A module owns its database tables. No other module may read or write them.
3. Cross-module data access happens via exported service methods, not SQL joins.
4. Shared types (Money, TenantID, BranchID) live in `pkg/` and are value objects.
5. If a module needs to react to another module's events, we use an in-process
   event bus (synchronous initially, async later if needed).

## Consequences

### Positive

- **Fast to ship**: One binary, one deployment pipeline, one database. No
  service mesh, no distributed tracing, no inter-service auth needed on day one.
- **Simple operations**: A single team member can deploy, monitor, and debug
  the entire backend. No Kubernetes required initially.
- **Refactoring safety**: Module boundaries enforce discipline. If we later
  need to extract a module (e.g., fiscal compliance) into a separate service,
  the interface is already defined.
- **Transactional consistency**: Cross-module operations can use database
  transactions when needed (e.g., creating an order and a payment record).
- **Testability**: Modules can be tested in isolation by mocking their
  dependency interfaces.

### Negative

- **Shared database coupling**: Schema migrations affect the entire system.
  A bad migration in one module can impact others.
- **Scaling granularity**: Cannot scale individual modules independently.
  The entire binary scales as one unit (horizontal scaling via multiple
  instances behind a load balancer).
- **Discipline required**: Module boundaries are enforced by convention and
  code review, not by the network. A lazy import can violate boundaries
  silently. Linting rules and CI checks must enforce this.

### Risks

- **Module boundary erosion**: Without automated enforcement, developers may
  take shortcuts and create cross-module dependencies. Mitigation: Go's
  `internal/` package visibility, custom linter rules, architecture tests
  in CI.
- **Database bottleneck**: All modules hitting one PostgreSQL instance could
  become a bottleneck at very high scale. Mitigation: PostgreSQL can handle
  the transaction volumes of hundreds of restaurant branches. This is a
  problem for 2028, not 2026.
- **Future extraction cost**: If we do need to extract a service, we will
  need to introduce network calls, handle partial failures, and potentially
  split the database. The modular structure makes this possible but not free.

## Alternatives Considered

### Microservices Architecture

Each domain as an independent service with its own database, communicating
via gRPC or message queues.

**Rejected because:**
- Team of 1-5 cannot maintain 8-10 services, each with its own deployment
  pipeline, monitoring, and scaling configuration.
- Distributed transactions across services (e.g., order creation + payment +
  fiscal signing) are extremely complex and error-prone.
- Operational overhead (service discovery, circuit breakers, distributed
  tracing) would consume more engineering time than feature development.
- The platform's current scale (tens of branches, not thousands) does not
  justify the complexity.

### Traditional (Unstructured) Monolith

Single codebase with no internal module boundaries. All code in a flat
structure with shared access to all database tables.

**Rejected because:**
- As the team grows, lack of boundaries leads to spaghetti code where
  every change risks breaking unrelated functionality.
- Makes future extraction to services nearly impossible without a
  rewrite.
- Testing becomes difficult as everything is coupled.
- Onboarding new developers is harder when there are no clear ownership
  boundaries.

### Serverless Functions (AWS Lambda / Cloud Functions)

Each API endpoint or domain operation as a serverless function.

**Rejected because:**
- Cold start latency is unacceptable for real-time POS sync operations.
- Vendor lock-in conflicts with our multi-cloud / self-hosted deployment
  goals.
- Complex state management and database connection pooling challenges.
- Debugging distributed serverless applications is significantly harder
  than debugging a single process.
