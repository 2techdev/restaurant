# ADR-013: Money Stored as Integer Cents

## Status
Accepted

## Date
2026-03-20

## Context

A restaurant POS system processes thousands of financial calculations daily:

- Line item totals: quantity x unit price
- Modifier price adjustments: +2.50 for extra cheese
- Discount calculations: 15% off, 5.00 fixed discount, buy-2-get-1-free
- Tax calculations: 7.7% MWST, 19% MwSt
- Payment splitting: divide 127.50 among 3 guests
- Rounding: Swiss 5 Rappen cash rounding
- Multi-currency: EUR and CHF in the same system

Financial calculations demand absolute precision. A rounding error of one
cent, compounded across thousands of transactions, can cause:

- Cash drawer discrepancies at shift close
- Tax reporting errors (auditors notice)
- Reconciliation failures with payment processors
- Customer trust issues (overcharging or undercharging)

### The Floating-Point Problem

IEEE 754 floating-point numbers (double, float) cannot exactly represent
most decimal fractions:

```
0.1 + 0.2 = 0.30000000000000004  (not 0.30)
19.99 * 100 = 1998.9999999999998  (not 1999)
2.50 + 2.50 + 2.50 = 7.5         (correct by luck)
10.00 / 3.0 = 3.3333333333333335  (inherent precision loss)
```

In a POS system, these errors accumulate. A restaurant processing 500
orders per day could accumulate rounding errors that make end-of-day
totals wrong by several cents or more.

## Decision

All monetary values throughout the entire platform (Branch Runtime SQLite,
Cloud Hub PostgreSQL, API payloads, ERPNext bridge) are stored and
calculated as **integers representing the smallest currency unit**.

### Currency Units

| Currency | Smallest Unit | Name      | Example               |
|----------|---------------|-----------|------------------------|
| CHF      | 1 Rappen      | Rappen    | CHF 15.90 = 1590       |
| EUR      | 1 Cent        | Cent      | EUR 15.90 = 1590       |

### Money Type

**Dart (Branch Runtime):**

```dart
/// Represents a monetary value in the smallest currency unit (cents/Rappen).
/// All arithmetic operates on integers. No floating-point involved.
class Money {
  final int cents;
  final CurrencyCode currency;

  const Money(this.cents, this.currency);

  /// Factory from major units (for display input parsing only)
  factory Money.fromMajor(double major, CurrencyCode currency) {
    return Money((major * 100).round(), currency);
  }

  Money operator +(Money other) {
    assert(currency == other.currency, 'Currency mismatch');
    return Money(cents + other.cents, currency);
  }

  Money operator -(Money other) {
    assert(currency == other.currency, 'Currency mismatch');
    return Money(cents - other.cents, currency);
  }

  /// Multiply by quantity (integer)
  Money operator *(int quantity) {
    return Money(cents * quantity, currency);
  }

  /// Apply percentage (e.g., 15% discount).
  /// Rounds half-up to nearest cent.
  Money percentage(int basisPoints) {
    // basisPoints: 1500 = 15.00%
    final result = (cents * basisPoints + 5000) ~/ 10000;
    return Money(result, currency);
  }

  /// Format for display: 1590 -> "15.90"
  String get display {
    final major = cents ~/ 100;
    final minor = (cents % 100).abs();
    return '$major.${minor.toString().padLeft(2, '0')}';
  }

  /// Format with currency: 1590 -> "CHF 15.90"
  String displayWithCurrency() => '${currency.code} $display';
}
```

**Go (Cloud Hub):**

```go
type Money struct {
    Cents    int64        `json:"cents"`
    Currency CurrencyCode `json:"currency"`
}

func (m Money) Add(other Money) Money {
    if m.Currency != other.Currency {
        panic("currency mismatch")
    }
    return Money{Cents: m.Cents + other.Cents, Currency: m.Currency}
}

func (m Money) Multiply(quantity int) Money {
    return Money{Cents: m.Cents * int64(quantity), Currency: m.Currency}
}

// Percentage applies a percentage and rounds half-up.
// basisPoints: 1500 = 15.00%, 725 = 7.25%
func (m Money) Percentage(basisPoints int) Money {
    result := (m.Cents*int64(basisPoints) + 5000) / 10000
    return Money{Cents: result, Currency: m.Currency}
}
```

### Calculation Rules

1. **All math is integer**: Addition, subtraction, multiplication by
   quantities -- all in cents.
2. **Percentages use basis points**: Tax rates and discount percentages
   are stored as basis points (integer). 19% = 1900 basis points.
   7.7% = 770 basis points.
3. **Rounding only once**: Rounding happens once at the final step
   (line total, tax amount, payment total), never in intermediate
   calculations.
4. **Half-up rounding**: Standard banker's rounding. 0.5 rounds up.
5. **Display is presentation-only**: The `display` method converts
   cents to a formatted string. This is a UI concern, never stored.

### Tax Calculation Example

```
Order:
  1x Schnitzel    CHF 28.50 (2850 cents)  @ 8.1% MWST
  2x Bier         CHF  6.50 (1300 cents each, 2600 total)  @ 8.1% MWST
  1x Dessert      CHF  9.00 (900 cents)   @ 2.6% MWST (reduced)

Line totals (integer math):
  Schnitzel:  2850 * 1 = 2850
  Bier:       1300 * 2 = 2600 (integer multiplication, no rounding needed)
  Dessert:    900 * 1  = 900

Tax per line (basis points):
  Schnitzel tax:  2850 * 810 / 10000 = 230.85 -> round to 231
  Bier tax:       2600 * 810 / 10000 = 210.60 -> round to 211
  Dessert tax:    900 * 260 / 10000  = 23.40  -> round to 23

Subtotal: 2850 + 2600 + 900 = 6350 (CHF 63.50)
Total tax: 231 + 211 + 23 = 465 (CHF 4.65)
Total:     6350 + 465 = 6815 (CHF 68.15)
```

### Swiss 5 Rappen Rounding

Switzerland rounds cash payment totals to the nearest 5 Rappen. This
rounding is applied **only to cash payments**, not to card payments.

```dart
/// Round to nearest 5 Rappen (Swiss cash rounding)
Money roundToFiveRappen() {
  assert(currency == CurrencyCode.CHF);
  final remainder = cents % 5;
  if (remainder == 0) return this;
  if (remainder >= 3) {
    return Money(cents + (5 - remainder), currency); // Round up
  }
  return Money(cents - remainder, currency);          // Round down
}
```

Examples:
- 6815 (CHF 68.15) -> 6815 (no change, already divisible by 5)
- 6817 (CHF 68.17) -> 6815 (round down, remainder 2)
- 6818 (CHF 68.18) -> 6820 (round up, remainder 3)
- 6813 (CHF 68.13) -> 6815 (round up, remainder 3)

The rounding difference is tracked as a separate line in the payment
record for accounting purposes.

## Consequences

### Positive

- **Exact arithmetic**: Integer math produces exact results. No
  floating-point surprises. 1590 + 1590 = 3180, always.
- **Cash drawer accuracy**: End-of-day totals match exactly. No
  accumulated rounding drift.
- **Cross-platform consistency**: The same integer calculation in Dart
  and Go produces identical results. No floating-point representation
  differences between platforms.
- **Database efficiency**: INTEGER columns are more compact and faster
  to index and compare than DECIMAL or NUMERIC types.
- **Audit confidence**: Financial records are exact integers. Auditors
  can verify calculations with simple integer math.
- **Tax compliance**: Tax authorities expect exact numbers. Integer
  cents with defined rounding rules produce deterministic, verifiable
  results.

### Negative

- **Developer discipline**: Every developer must understand that monetary
  values are in cents, not in major units. A bug that stores 15.90 as
  the integer 15 instead of 1590 causes a 100x error. Mitigation: the
  Money type enforces this -- raw integers are only used in the database
  layer, never in business logic.
- **Display conversion**: Every UI component must convert cents to display
  format. This is repetitive but straightforward.
- **Third-party integration**: External APIs may use decimal strings or
  floats for money. Conversion must happen at the integration boundary,
  with explicit rounding.

### Risks

- **Integer overflow**: Dart's `int` is 64-bit (max ~9.2 quintillion
  cents, or ~92 quadrillion CHF). Overflow is not a practical concern
  for restaurant POS. PostgreSQL `INTEGER` (32-bit) maxes at ~21 million
  CHF per field, which is sufficient. If needed, use `BIGINT`.
- **Multi-currency calculations**: Converting between EUR and CHF
  requires exchange rates that are inherently fractional. Mitigation:
  currency conversion happens at a defined boundary with explicit
  rounding rules. The system does not support mixed-currency orders.
- **Basis point precision**: Some tax rates require more than 2 decimal
  places (e.g., 7.7% Swiss MWST = 770 basis points). Basis points
  (4 decimal places, 10000 = 100%) provide sufficient precision for
  all European tax rates.

## Alternatives Considered

### Floating-Point (double / float)

Store money as `double` or `float` values representing major currency
units (15.90).

**Rejected because:**
- IEEE 754 floating-point cannot exactly represent most decimal values.
- Accumulated rounding errors cause financial discrepancies.
- Different platforms (Dart VM, Go, SQLite, PostgreSQL) may round
  floating-point differently, causing cross-platform inconsistencies.
- Industry best practice unanimously recommends against floating-point
  for money.

### Decimal / BigDecimal Types

Use language-specific decimal types (Python's `Decimal`, Java's
`BigDecimal`) or PostgreSQL's `NUMERIC` type.

**Not fully rejected** -- PostgreSQL `NUMERIC` could be used on the
cloud side. However:
- Dart does not have a built-in `Decimal` type. Third-party packages
  exist but add dependency and complexity.
- SQLite has no native `DECIMAL` type.
- Integer cents are simpler, faster, and universally supported across
  all our platforms (Dart, Go, SQLite, PostgreSQL).
- For POS calculations (addition, multiplication by quantity, percentage
  application), integer cents with defined rounding rules are sufficient
  and more predictable than arbitrary-precision decimals.

### String-Based Decimal Representation

Store money as strings ("15.90") and parse for calculations.

**Rejected because:**
- String parsing on every calculation is slow and error-prone.
- No compile-time type safety for arithmetic operations.
- Sorting and comparison require parsing or specialized collation.
- This approach is sometimes used for serialization (API payloads)
  but is not suitable for storage or computation.
