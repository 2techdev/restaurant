# ADR-015: Soft Delete with Tombstones, Never Physical Delete

## Status
Accepted

## Date
2026-03-20

## Context

A restaurant POS platform accumulates data over time: orders, payments,
products, users, tables, shifts, receipts, fiscal records. The question
of how to handle data deletion arises in several scenarios:

- A product is discontinued (removed from the menu)
- An employee leaves the restaurant
- A table layout changes (table removed)
- A category is merged with another
- Test/training data needs to be cleaned up
- GDPR data subject requests deletion of personal data

Simultaneously, strict data retention requirements apply:

### German Legal Requirements
- **GoBD**: All financially relevant records must be retained for 10 years
  in unaltered form. This includes orders, payments, receipts, and related
  master data.
- **KassenSichV / DSFinV-K**: TSE-signed transactions must be retained and
  reproducible for tax audit for 10 years.
- **AO (Abgabenordnung)**: General tax record retention of 10 years for
  accounting records, 6 years for business correspondence.

### Swiss Legal Requirements
- **OR (Obligationenrecht) Art. 958f**: Business records must be retained
  for 10 years.
- **MWST**: VAT records retained for 10 years.

### Sync Requirements
- Deletion must be propagated to all devices in the branch (via LAN sync)
  and to the cloud (via cloud sync).
- If a record is physically deleted on one device, other devices that
  still reference it will have dangling references (broken foreign keys,
  missing product names in order history).
- Sync depends on knowing about every change, including deletions. A
  physical delete leaves no trace for the sync engine to detect and
  propagate.

### Audit and Recovery
- Accidental deletions must be recoverable. A waiter accidentally deleting
  a product or a manager deleting the wrong category should not cause
  permanent data loss.
- Audit trails require knowing when something was deleted and by whom.

## Decision

All entities in the system use **soft delete** with tombstone propagation.
Physical deletion never occurs in application code.

### Soft Delete Schema

Every table includes soft delete columns:

```sql
-- PostgreSQL (Cloud Hub)
CREATE TABLE products (
    id          UUID PRIMARY KEY,
    name        TEXT NOT NULL,
    -- ... other fields ...
    is_deleted  BOOLEAN NOT NULL DEFAULT false,
    deleted_at  TIMESTAMP,
    deleted_by  UUID REFERENCES users(id),
    version     INTEGER NOT NULL DEFAULT 1,
    created_at  TIMESTAMP NOT NULL,
    updated_at  TIMESTAMP NOT NULL
);

-- Partial index: only non-deleted records for fast queries
CREATE INDEX idx_products_active ON products (id) WHERE NOT is_deleted;
```

```dart
// Drift (Branch Runtime SQLite)
class Products extends Table {
  TextColumn get id => text()();
  TextColumn get name => text()();
  // ... other fields ...
  BoolColumn get isDeleted => boolean().withDefault(const Constant(false))();
  DateTimeColumn get deletedAt => dateTime().nullable()();
  TextColumn get deletedBy => text().nullable()();
  IntColumn get version => integer().withDefault(const Constant(1))();
  DateTimeColumn get createdAt => dateTime()();
  DateTimeColumn get updatedAt => dateTime()();

  @override
  Set<Column> get primaryKey => {id};
}
```

### Delete Operation

When a record is "deleted":

```dart
Future<void> softDeleteProduct(String productId, String deletedByUserId) async {
  await (update(products)
    ..where((p) => p.id.equals(productId))
  ).write(ProductsCompanion(
    isDeleted: const Value(true),
    deletedAt: Value(DateTime.now()),
    deletedBy: Value(deletedByUserId),
    version: products.version + const Constant(1),
    updatedAt: Value(DateTime.now()),
  ));
}
```

The record remains in the database. The `is_deleted` flag marks it as
logically deleted. The `version` increment triggers sync propagation.

### Query Patterns

**Active records (default):**
```dart
// All queries filter out deleted records by default
Future<List<Product>> getActiveProducts() {
  return (select(products)
    ..where((p) => p.isDeleted.equals(false))
    ..orderBy([(p) => OrderingTerm.asc(p.name)])
  ).get();
}
```

**Including deleted (for admin/audit views):**
```dart
// Explicit opt-in to see deleted records
Future<List<Product>> getAllProductsIncludingDeleted() {
  return (select(products)
    ..orderBy([(p) => OrderingTerm.asc(p.name)])
  ).get();
}
```

**Historical reference (for order display):**
```dart
// When displaying old orders, the referenced product may be soft-deleted.
// The order line contains a snapshot (ADR-010), but we may still want
// to look up the product for additional context.
Future<Product?> getProductIncludingDeleted(String id) {
  return (select(products)..where((p) => p.id.equals(id))).getSingleOrNull();
}
```

### Tombstone Propagation via Sync

When a record is soft-deleted:

1. The `version` is incremented, triggering the sync engine.
2. **LAN sync** broadcasts the change to all devices in the branch.
   Each device updates its local copy: `is_deleted = true`.
3. **Cloud sync** pushes the change to the Cloud Hub. The Cloud Hub
   updates its PostgreSQL record.
4. Other branches (if the record is branch-global, like a product)
   receive the deletion via cloud sync on their next pull.

The tombstone (soft-deleted record) exists on all devices. No device
has a dangling reference.

### Archival Strategy

Soft-deleted records accumulate over time. To manage storage:

**After 2 years**: Automated archival job moves soft-deleted records to
archive tables in PostgreSQL. Archive tables have the same schema but are
stored on cheaper storage (or compressed tablespace).

```sql
-- Archive table
CREATE TABLE archived_orders (
    LIKE orders INCLUDING ALL
);

-- Archival job (runs monthly)
INSERT INTO archived_orders
SELECT * FROM orders
WHERE is_deleted = true
  AND deleted_at < NOW() - INTERVAL '2 years';

DELETE FROM orders
WHERE is_deleted = true
  AND deleted_at < NOW() - INTERVAL '2 years'
  AND id IN (SELECT id FROM archived_orders);
```

Archive tables remain queryable for audit purposes. They are simply
separated from the active dataset for performance.

**On mobile devices**: Completed and soft-deleted records older than
6 months are removed from the local SQLite database to save storage.
The cloud retains the full history.

### Physical Deletion Rules

Physical deletion is permitted only under specific conditions:

| Scenario                           | Physical Delete? | Authority          |
|------------------------------------|------------------|--------------------|
| Application code (any feature)     | NEVER            | --                 |
| Admin tool (manual cleanup)        | NEVER            | --                 |
| Automated archival (2+ years)      | Cloud only       | Scheduled job      |
| Non-fiscal data (3+ years)         | Allowed          | Automated job only |
| Fiscal data (orders, payments)     | NEVER            | 10-year retention  |
| GDPR erasure request               | Special process  | See below          |
| Device storage cleanup (6+ months) | Device only      | Automated job      |

### GDPR Right to Erasure

GDPR Article 17 grants individuals the right to request deletion of their
personal data. This creates a tension with fiscal retention requirements.

**Resolution:**
- **Customer personal data** (name, email, phone, address): Can be
  pseudonymized after the business relationship ends. Replace with
  `"[REDACTED]"` while keeping the record structure intact.
- **Employee personal data**: Same pseudonymization approach after
  employment ends and legal retention period for employment records
  expires.
- **Transaction data**: Cannot be deleted or pseudonymized during the
  10-year fiscal retention period. After 10 years, personal data within
  transactions can be pseudonymized.
- **Legal basis**: Fiscal retention obligations (GoBD, OR Art. 958f)
  provide legal basis under GDPR Article 17(3)(b) for retaining
  transaction records despite erasure requests.

```sql
-- GDPR pseudonymization (not deletion)
UPDATE customers
SET
    first_name = '[REDACTED]',
    last_name = '[REDACTED]',
    email = '[REDACTED]',
    phone = '[REDACTED]',
    address = '[REDACTED]',
    gdpr_erased_at = NOW(),
    version = version + 1
WHERE id = 'customer-uuid'
  AND is_deleted = true;
```

## Consequences

### Positive

- **Legal compliance**: Fiscal records are never lost. 10-year retention
  requirements are satisfied by design. Auditors can access complete
  transaction history.
- **Sync reliability**: Tombstones propagate like any other change.
  No special handling for deletions. No dangling references across
  devices.
- **Undo capability**: Soft-deleted records can be restored by flipping
  `is_deleted` back to `false`. Accidental deletions are recoverable.
- **Audit trail**: `deleted_at` and `deleted_by` provide full
  accountability for every deletion.
- **Referential integrity**: Order lines can always look up their
  associated product (even if discontinued). Historical reports remain
  accurate.
- **Simplicity**: No cascading delete logic. No orphan cleanup. No
  "is this record referenced somewhere?" checks before deletion.

### Negative

- **Storage growth**: Data accumulates forever (until archival).
  Mitigation: POS data is small. A busy restaurant generates roughly
  100MB per year of transactional data. Storage costs are negligible
  compared to the risks of data loss.
- **Query overhead**: Every query must include `WHERE is_deleted = false`.
  Mitigation: partial indexes (`WHERE NOT is_deleted`) in PostgreSQL
  ensure index scans only cover active records. Drift query helpers
  add the filter automatically.
- **Index bloat**: Soft-deleted records remain in regular indexes
  (unless partial indexes are used). Mitigation: partial indexes on
  PostgreSQL. On SQLite, the volume of soft-deleted records is small
  enough (device-level cleanup after 6 months) that impact is minimal.
- **Complexity for aggregations**: Reporting queries must be careful
  to exclude soft-deleted records from counts and sums (unless the
  report specifically covers historical data including deletions).

### Risks

- **Forgotten filter**: A developer forgets the `is_deleted = false`
  filter in a query, accidentally including deleted records in results.
  Mitigation: create a default query scope/view that excludes deleted
  records. Make including deleted records an explicit opt-in.
  Code review checklist includes this check.
- **GDPR conflict**: A data subject insists on physical deletion, not
  pseudonymization. Mitigation: legal counsel confirms that fiscal
  retention obligations override GDPR erasure for transaction records.
  Pseudonymization is the accepted approach under GDPR Article 17(3)(b).
- **Archival job failure**: If the archival job fails, old data
  accumulates and may degrade query performance on the cloud database.
  Mitigation: monitoring and alerting on archival job execution.
  Performance impact is minimal until tens of millions of records
  accumulate (years of operation for many branches).
- **Device storage exhaustion**: If the on-device cleanup job fails,
  old records consume tablet storage. Mitigation: monitoring of device
  storage levels, alerts when storage drops below threshold, manual
  cleanup tools in admin UI.

## Alternatives Considered

### Physical Delete with Audit Log

Physically delete records but log every deletion in a separate audit table.

**Rejected because:**
- Violates GoBD requirements (original records must be preserved).
- Audit table is a secondary record, not the original. Auditors may
  question its authenticity.
- Referential integrity is broken when referenced records are physically
  deleted.
- Sync cannot propagate a deletion if the record no longer exists.
  Would need a separate "deletion event" sync mechanism.
- Recovery from accidental deletion requires rebuilding from audit log.

### Hard Delete with Cascade

Physical delete with cascading foreign key deletes.

**Rejected because:**
- Catastrophic risk: deleting a category could cascade-delete all
  products in that category, and all order lines referencing those
  products.
- Impossible to undo.
- Violates every data retention requirement.
- Sync cannot propagate cascading deletes reliably.

### Separate Archive Database from Day One

Move all "deleted" records to a separate archive database immediately
on deletion.

**Rejected because:**
- Adds operational complexity (two databases to manage, query, and
  back up) from the start.
- Querying across active and archived data (for reports covering long
  time periods) requires cross-database queries.
- Simple soft delete achieves the same goal with less infrastructure.
- Archival can be introduced later when data volume justifies it
  (the 2-year archival strategy handles this).

### Event Sourcing for Deletion Tracking

Use event sourcing to record deletion events, deriving current state
from the event log.

**Rejected because:**
- Full event sourcing is overly complex for deletion tracking
  (see ADR-010 discussion).
- A simple `is_deleted` flag achieves the same outcome with far less
  complexity.
- Event sourcing would require projections for every query, adding
  latency and infrastructure.
