# ADR-014: LAN Sync for Intra-Branch, Cloud Sync for Persistence

## Status
Accepted

## Date
2026-03-20

## Context

A typical restaurant branch has multiple devices that must stay coordinated:

- **POS Terminal 1** (register): Takes counter orders, processes payments
- **POS Terminal 2** (register): Second register for peak hours
- **Waiter Tablet 1-3**: Table-side ordering
- **KDS Screen 1** (kitchen): Displays food preparation orders
- **KDS Screen 2** (bar): Displays drink preparation orders

These devices must share a consistent view of:
- **Order state**: When Waiter Tablet 1 creates an order, KDS Screen 1
  must display it within seconds.
- **Table status**: When POS Terminal 1 marks Table 7 as occupied, all
  waiter tablets must reflect this immediately.
- **Payment status**: When a payment is processed, all devices must know
  the order is paid.
- **Product availability**: When the kitchen marks an item as 86'd
  (out of stock), all ordering devices must stop offering it.

Simultaneously, the Cloud Hub needs to receive all transaction data for:
- Multi-branch reporting and analytics
- Fiscal compliance (TSE signing via Fiskaly -- ADR-008)
- ERPNext accounting bridge (ADR-003)
- Backup and disaster recovery
- License management and feature flag updates

Two distinct sync problems exist:
1. **Intra-branch real-time**: Devices in the same restaurant, same network,
   need sub-second updates. This must work without internet.
2. **Branch-to-cloud persistence**: Transaction data must reach the cloud
   for reporting, compliance, and backup. This can tolerate latency.

## Decision

We implement **two separate sync mechanisms** optimized for their
respective use cases:

1. **LAN Sync**: mDNS discovery + WebSocket for intra-branch real-time
2. **Cloud Sync**: HTTPS REST for branch-to-cloud persistence

### LAN Sync Architecture

```
Branch Network (WiFi / LAN)
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Lead Device (elected)                                  │
│  ┌──────────────────┐                                   │
│  │ POS Terminal 1   │ <-- mDNS broadcast: "_pos._tcp"   │
│  │ WebSocket Server │                                   │
│  │ SQLite (primary) │                                   │
│  └──────┬───────────┘                                   │
│         │ WebSocket connections                         │
│    ┌────┼────────┬──────────┬──────────┐                │
│    │    │        │          │          │                │
│  ┌─┴──┐ ┌─┴──┐ ┌──┴──┐ ┌───┴──┐ ┌───┴──┐             │
│  │POS2│ │WT1 │ │WT2  │ │KDS1  │ │KDS2  │             │
│  └────┘ └────┘ └─────┘ └──────┘ └──────┘             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Discovery**: Devices discover each other via **mDNS** (multicast DNS,
also known as Bonjour/Avahi). The lead device advertises a service
`_pos._tcp.local` on the local network. Other devices discover it and
connect via WebSocket.

**Lead Election**: The first device to start (or the device configured
as "lead" in branch settings) becomes the lead. If the lead goes offline,
another device can be promoted. The lead device runs the WebSocket server
that coordinates sync.

**Protocol**: WebSocket with JSON messages.

```json
// Device -> Lead: I changed something
{
  "type": "sync_push",
  "device_id": "device-uuid",
  "changes": [
    {
      "table": "orders",
      "id": "order-uuid",
      "version": 42,
      "operation": "upsert",
      "data": { ... }
    }
  ]
}

// Lead -> All Devices: Broadcast changes
{
  "type": "sync_broadcast",
  "source_device": "device-uuid",
  "changes": [
    {
      "table": "orders",
      "id": "order-uuid",
      "version": 42,
      "operation": "upsert",
      "data": { ... }
    }
  ]
}
```

**Conflict Resolution (LAN)**:
- **Last-Writer-Wins (LWW)** for most fields: The change with the higher
  `version` number wins. `version` is incremented on every modification.
- **Merge for line items**: Order line additions from different devices
  are merged (union). Line removals require explicit void operations.
- **Deterministic tiebreaker**: If two changes have the same version,
  the device with the lexicographically higher device ID wins (arbitrary
  but deterministic).

### Cloud Sync Architecture

```
Branch Runtime                    Cloud Hub
┌──────────────┐                 ┌──────────────────┐
│              │   HTTPS REST    │                  │
│ Sync Engine  │ ──────────────> │ Sync API         │
│              │                 │                  │
│ Change Queue │ <────────────── │ Change Feed      │
│ (SQLite)     │                 │ (PostgreSQL)     │
└──────────────┘                 └──────────────────┘
```

**Protocol**: HTTPS REST with JSON payloads. Batched operations.

**Push (Device -> Cloud):**
```
POST /api/v1/sync/push
Authorization: Bearer <device-token>

{
  "device_id": "device-uuid",
  "branch_id": "branch-uuid",
  "last_sync_version": 1234,
  "changes": [
    { "table": "orders", "id": "...", "version": 42, "data": {...} },
    { "table": "payments", "id": "...", "version": 1, "data": {...} }
  ]
}
```

**Pull (Cloud -> Device):**
```
GET /api/v1/sync/pull?since_version=1234&limit=100
Authorization: Bearer <device-token>

Response:
{
  "changes": [ ... ],
  "has_more": true,
  "next_version": 1334
}
```

**Sync Strategy:**
1. Device maintains a `last_sync_version` (monotonically increasing
   cloud version number).
2. On sync: push local changes first, then pull remote changes.
3. Push includes all locally modified records since last successful push.
4. Pull includes all cloud changes since `last_sync_version`.
5. Sync runs periodically (every 30 seconds when idle, immediately
   after order completion) and on connectivity restoration.

**Conflict Resolution (Cloud)**:
- Same LWW strategy as LAN sync.
- Cloud is the authoritative tiebreaker for cross-branch conflicts.
- Immutable transaction records (ADR-010) never conflict -- they are
  append-only.

### Sync Priorities

Not all data has the same sync urgency:

| Data Type          | LAN Sync | Cloud Sync | Priority |
|--------------------|----------|------------|----------|
| Active orders      | Immediate| On change  | P0       |
| Table status       | Immediate| Batch      | P0       |
| KDS updates        | Immediate| Not synced | P0       |
| Payments           | Immediate| On change  | P0       |
| Product 86'd       | Immediate| Batch      | P1       |
| Completed orders   | N/A      | Batch      | P1       |
| License token      | N/A      | On connect | P0       |
| Catalog changes    | N/A      | On connect | P2       |
| Shift reports      | N/A      | End of day | P2       |
| Fiscal data        | N/A      | On change  | P0       |

### Fallback Behavior

```
Primary:   LAN Sync (fast, local)
Fallback:  Cloud Sync (reliable, remote)

If LAN sync fails (device not on same network):
  -> Use cloud sync for all coordination (higher latency but functional)

If cloud sync fails (no internet):
  -> LAN sync continues unaffected
  -> Local changes queue for cloud push when internet returns
  -> Queue persisted in SQLite (survives app restart)

If both fail (isolated device):
  -> Device operates independently
  -> All changes queued for sync
  -> Potential for conflicts resolved on reconnection
```

## Consequences

### Positive

- **Sub-second local updates**: KDS sees new orders within milliseconds
  via LAN WebSocket. No cloud round-trip needed.
- **Offline resilience**: LAN sync works without internet. Multiple
  devices in a branch coordinate via local network only.
- **Efficient cloud usage**: Cloud sync is batched and compressed,
  reducing bandwidth and server load. Not every keystroke goes to
  the cloud.
- **Independent scaling**: LAN sync scales with local network
  bandwidth (practically unlimited for POS data). Cloud sync scales
  with internet bandwidth and server capacity independently.
- **Clear separation of concerns**: LAN sync handles real-time UX.
  Cloud sync handles persistence, reporting, and compliance.

### Negative

- **Two sync engines**: Two separate sync mechanisms to build, test,
  and maintain. Different protocols, different conflict resolution
  edge cases, different failure modes.
- **Consistency lag**: A brief window exists where LAN-synced data
  has not yet reached the cloud. Cloud reports may lag behind real-time
  branch state.
- **Lead device dependency**: LAN sync depends on the lead device
  being available. If the lead crashes, LAN sync is interrupted until
  a new lead is elected or the device recovers.
- **Network configuration**: mDNS requires multicast to work on the
  local network. Some enterprise WiFi configurations block multicast.
  Mitigation: fallback to direct IP configuration.

### Risks

- **Split-brain in LAN**: If the local network partitions (devices on
  different VLANs or access points), two groups may diverge. Mitigation:
  cloud sync reconciles all devices eventually. Split-brain is rare
  in typical restaurant WiFi setups.
- **Change queue overflow**: If a device is offline for extended periods,
  the change queue grows. Mitigation: queue is persisted in SQLite with
  configurable size limits. Old non-critical changes can be dropped in
  favor of recent ones.
- **Version clock drift**: If version numbers get out of sync between
  devices, conflict resolution may produce unexpected results.
  Mitigation: versions are per-record, not global. Each record's version
  is independent and deterministic.
- **WebSocket connection limits**: A branch with many devices could
  overwhelm the lead device's WebSocket capacity. Mitigation: typical
  branches have 3-8 devices. Flutter/Dart WebSocket handling is
  efficient for this scale.

## Alternatives Considered

### Cloud-Only Sync (No LAN)

All device coordination goes through the cloud.

**Rejected because:**
- Adds 100-500ms latency to every operation (network round-trip to cloud).
  KDS updates would be noticeably delayed.
- Completely non-functional when internet is down. Devices in the same
  room cannot coordinate.
- Unnecessary cloud load for intra-branch coordination.
- Bandwidth cost for real-time updates.

### Peer-to-Peer Sync (No Lead Device)

Every device syncs directly with every other device (mesh network).

**Rejected because:**
- O(n^2) connections for n devices. 6 devices = 15 connections.
- Conflict resolution without a coordinator is significantly more
  complex (vector clocks, CRDTs).
- Difficult to determine authoritative state without a leader.
- Lead-based star topology is simpler and sufficient for 3-8 devices.

### Firebase Realtime Database / Firestore

Use Google's real-time database as the sync mechanism.

**Rejected because:**
- Vendor lock-in to Google Cloud Platform.
- No LAN-only mode -- requires internet for all sync.
- Pricing based on reads/writes could be expensive for high-frequency
  POS operations.
- Data stored on Google servers conflicts with European data sovereignty
  requirements for some customers.
- Cannot self-host.

### CRDTs (Conflict-Free Replicated Data Types)

Use CRDTs for automatic conflict resolution without coordination.

**Not fully rejected** -- CRDTs may be adopted for specific data types
in the future (e.g., concurrent modifier selection on the same order).
However:
- CRDTs add significant complexity to the data model.
- Most POS operations have clear ownership (one device modifies an
  order at a time) and do not need CRDT-level conflict resolution.
- LWW with version numbers is sufficient for current requirements.
- CRDTs can be introduced incrementally for specific conflict-prone
  fields without a full architectural change.
