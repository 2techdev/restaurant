# ADR-012: Country Packs as Pluggable Modules

## Status
Accepted

## Date
2026-03-20

## Context

The restaurant POS platform launches in Switzerland and Germany, then
expands to other European markets. Each country has distinct requirements
in several areas:

### Fiscal Compliance
- **Germany**: TSE requirement (KassenSichV), DSFinV-K export format,
  Finanzamt audit requirements, GoBD data retention.
- **Switzerland**: No TSE requirement, but VAT reporting obligations,
  MWST (Mehrwertsteuer) rules with three rates (8.1%, 2.6%, 3.8%).
- **Austria**: Registrierkassenpflicht (cash register obligation),
  A-Trust or similar fiscal signing, RKSV compliance.
- **France**: NF 525 certification for POS software, JCD (Journal
  Comptable Detaille) export.

### Tax Calculation
- **Germany**: 7% reduced / 19% standard VAT. In-restaurant vs.
  takeaway changes the rate for food.
- **Switzerland**: 2.6% reduced / 3.8% accommodation / 8.1% standard
  MWST. 5 Rappen rounding for cash payments.
- **Austria**: 10% reduced / 13% intermediate / 20% standard.
- **France**: 5.5% reduced / 10% intermediate / 20% standard.

### Receipt and Invoice Formatting
- **Germany**: TSE signature block, QR code with fiscal data, specific
  legal text requirements.
- **Switzerland**: QR-bill format for invoices (ISO 20022), MWST number
  on receipts.
- **Austria**: QR code with RKSV signature, Kassenidentifikationsnummer.
- **France**: NF 525 certification number, specific mention requirements.

### Payment and Banking
- **Switzerland**: QR-bill (QR-Rechnung) replacing ESR payment slips.
  PostFinance integration.
- **Germany**: SEPA direct debit common. EC-Karte (Girocard) as primary
  card payment method.
- **SEPA zone**: Shared SEPA credit transfer and direct debit standards.

Hardcoding all of this into the core system would create an unmaintainable
tangle of if-else branches and country-specific logic scattered throughout
the codebase.

## Decision

We implement country-specific logic as **pluggable Country Pack modules**
that register with the core system via well-defined interfaces.

### Interface Definitions

The core system defines interfaces (Go interfaces on Cloud Hub, Dart
abstract classes on Branch Runtime) that Country Packs must implement:

```go
// Cloud Hub interfaces (Go)

type TaxCalculator interface {
    // Calculate tax for a set of line items
    CalculateLineTax(item TaxableItem) TaxResult
    // Apply country-specific rounding to a payment total
    RoundPaymentTotal(cents int, paymentMethod PaymentMethod) int
    // Get available tax rates for this country
    AvailableTaxRates() []TaxRate
}

type FiscalProvider interface {
    // Sign a completed transaction
    SignTransaction(ctx context.Context, tx FiscalTransaction) (*FiscalSignature, error)
    // Generate fiscal export for audit
    GenerateExport(ctx context.Context, params ExportParams) (io.Reader, error)
    // Validate fiscal compliance of a transaction
    ValidateCompliance(tx FiscalTransaction) []ComplianceIssue
}

type ReceiptFormatter interface {
    // Format receipt with country-specific requirements
    FormatReceipt(order Order, payment Payment, fiscal *FiscalSignature) ReceiptData
    // Get required legal text for receipts
    RequiredReceiptText() []string
}

type InvoiceGenerator interface {
    // Generate a country-compliant invoice
    GenerateInvoice(order Order, customer Customer) InvoiceData
    // Format payment reference (e.g., QR-bill for Switzerland)
    FormatPaymentReference(invoice InvoiceData) PaymentReference
}

type BankingAdapter interface {
    // Generate payment file (e.g., SEPA XML, QR-bill)
    GeneratePaymentFile(payments []PaymentRecord) ([]byte, error)
    // Parse bank statement for reconciliation
    ParseBankStatement(data []byte) ([]BankTransaction, error)
}
```

### Country Pack Structure

```
internal/
  countrypack/
    registry.go          -- Country pack registration and lookup
    interfaces.go        -- Interface definitions
    de/                  -- Germany pack
      tax.go             -- German VAT calculation (7%/19%, in/out rules)
      fiscal.go          -- Fiskaly TSE integration
      dsfinvk.go         -- DSFinV-K export generation
      receipt.go         -- German receipt formatting (TSE block, QR)
      invoice.go         -- German invoice requirements
      banking.go         -- SEPA/Girocard specifics
      pack.go            -- Pack registration entry point
    ch/                  -- Switzerland pack
      tax.go             -- Swiss MWST calculation (2.6%/3.8%/8.1%)
      rounding.go        -- 5 Rappen cash rounding
      receipt.go         -- Swiss receipt formatting
      invoice.go         -- QR-bill generation (ISO 20022)
      banking.go         -- PostFinance/QR-bill payment
      pack.go            -- Pack registration entry point
    at/                  -- Austria pack (future)
    fr/                  -- France pack (future)
```

### Registration Pattern

```go
// registry.go
type CountryPackRegistry struct {
    packs map[CountryCode]CountryPack
}

type CountryPack struct {
    Code             CountryCode
    Name             string
    TaxCalculator    TaxCalculator
    FiscalProvider   FiscalProvider      // nil if country has no fiscal req
    ReceiptFormatter ReceiptFormatter
    InvoiceGenerator InvoiceGenerator
    BankingAdapter   BankingAdapter
}

func (r *CountryPackRegistry) Register(pack CountryPack) {
    r.packs[pack.Code] = pack
}

func (r *CountryPackRegistry) Get(code CountryCode) (CountryPack, error) {
    pack, ok := r.packs[code]
    if !ok {
        return CountryPack{}, fmt.Errorf("no country pack for %s", code)
    }
    return pack, nil
}
```

```go
// de/pack.go
func init() {
    registry.Register(CountryPack{
        Code:             "DE",
        Name:             "Germany",
        TaxCalculator:    &GermanTaxCalculator{},
        FiscalProvider:   &FiskalyProvider{},
        ReceiptFormatter: &GermanReceiptFormatter{},
        InvoiceGenerator: &GermanInvoiceGenerator{},
        BankingAdapter:   &SEPAAdapter{},
    })
}
```

### Branch Runtime Mirroring

The Flutter client mirrors the Country Pack pattern:

```dart
abstract class TaxCalculator {
  TaxResult calculateLineTax(TaxableItem item);
  int roundPaymentTotal(int cents, PaymentMethod method);
  List<TaxRate> availableTaxRates();
}

class GermanTaxCalculator implements TaxCalculator {
  // 7% for food (in-house and takeaway)
  // 19% for beverages (in-house), 19% for non-food
  // Takeaway food: 7%, Dine-in food: 19% (pre-2024 rules vary)
}

class SwissTaxCalculator implements TaxCalculator {
  // 8.1% standard, 2.6% reduced, 3.8% accommodation
  // 5 Rappen rounding for cash payments only
}
```

## Consequences

### Positive

- **Clean separation**: Core POS logic knows nothing about German TSE
  or Swiss Rappen rounding. It calls `taxCalculator.CalculateLineTax()`
  and gets the right answer for the configured country.
- **Independent development**: The Germany pack and Switzerland pack can
  be developed and tested independently. A bug fix in German fiscal
  signing does not risk breaking Swiss tax calculation.
- **Easy expansion**: Adding Austria means implementing the interfaces
  for Austrian rules and registering a new pack. No changes to core
  POS code.
- **Testability**: Each Country Pack is tested against its own country's
  rules with country-specific test cases. Core logic is tested with
  mock implementations.
- **Regulatory agility**: When Germany changes TSE rules or Switzerland
  updates MWST rates, only the affected pack changes. The blast radius
  is contained.

### Negative

- **Interface design challenge**: The interfaces must be general enough
  to accommodate all countries but specific enough to be useful. This
  requires understanding requirements across multiple jurisdictions
  upfront.
- **Duplication risk**: Some logic is similar across countries (e.g.,
  SEPA is used in both Germany and Switzerland). Must identify shared
  components and extract them without over-abstracting.
- **Two implementations**: Each Country Pack exists in both Go (Cloud Hub)
  and Dart (Branch Runtime). Tax calculation logic, in particular, must
  produce identical results in both languages.
- **Testing burden**: Each Country Pack needs its own comprehensive test
  suite with real-world test cases from that country's tax and fiscal
  regulations.

### Risks

- **Interface instability**: If early Country Packs drive the interface
  design, later countries may require interface changes that break
  existing packs. Mitigation: design interfaces based on a survey of
  requirements across DE, CH, AT, and FR before finalizing. Accept that
  some evolution is inevitable and plan for versioned interfaces.
- **Regulatory complexity**: Some countries have rules that do not fit
  neatly into the defined interfaces (e.g., French NF 525 certification
  requires the entire POS software to be certified, not just a module).
  Mitigation: Country Pack interfaces include a `ValidateCompliance()`
  method for country-specific validation that can flag system-level
  requirements.
- **Tax calculation divergence**: If Go and Dart implementations of
  the same Country Pack produce different results, financial data will
  be inconsistent between device and cloud. Mitigation: shared test
  cases (JSON-based) run against both implementations in CI.

## Alternatives Considered

### Hardcoded Country Logic in Core

Country-specific if-else branches throughout the codebase.

**Rejected because:**
- Every new country adds complexity to every file that touches tax,
  receipts, or fiscal compliance.
- High risk of breaking one country when modifying another.
- Impossible to test one country in isolation.
- Codebase becomes unreadable as country count grows.

### Configuration-Driven Rules Engine

Define all country rules as configuration (JSON/YAML) interpreted by
a generic rules engine.

**Rejected because:**
- Fiscal compliance logic (TSE signing, DSFinV-K export) is too complex
  for configuration. It requires real code with API calls, error handling,
  and business logic.
- A rules engine is another piece of infrastructure to build and maintain.
- Debugging configuration-driven logic is harder than debugging code.
- Tax calculation edge cases (German in-house vs. takeaway rates) are
  easier to express in code than configuration.

### Third-Party Tax/Fiscal Service for All Countries

Use a SaaS provider (Avalara, Vertex, TaxJar) for tax calculation and
compliance.

**Rejected because:**
- Adds a runtime dependency that violates offline-first (ADR-002).
  Tax calculation must work offline.
- Ongoing per-transaction costs that scale with order volume.
- Limited coverage of European fiscal requirements (most are US-focused).
- Loss of control over critical compliance logic.
- We still need country-specific receipt formatting and invoice
  generation, which these services do not provide.

### Separate Application per Country

Build and deploy a different version of the POS for each country.

**Rejected because:**
- Forks diverge quickly. Bug fixes must be ported to every fork.
- Core POS features (ordering, kitchen, payments) are identical across
  countries. Duplicating them is wasteful.
- Customers operating in multiple countries (Swiss chain expanding to
  Germany) would need different applications.
- Unscalable beyond 2-3 countries.
